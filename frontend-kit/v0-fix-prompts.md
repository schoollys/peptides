# PeptideTrust — v0 FIX Prompts (UX-аудит, раунд 2)

Назначение: заготовка под реальную разработку. Промпты на русском, готовы к вставке в v0.app по одному.
Базовый деплой аудита: https://peptidetrust-app-setup.vercel.app
Дизайн-токены сохранять как есть: основной `#533afd`, тени `shadow-stripe-*`, tabular-nums, тиры `tier-gold #c9a227 / tier-silver #9aa0a6 / tier-bronze #b07a45 / tier-new(provisional) #5b78a8`, error `#b3261e`.

Источники истины: `reports/design-information-architecture.html` (IA/sitemap/states), `reports/design-wireframes.html` (экраны), `reports/design-system.html` (токены/состояния), `reports/spec-algorithm.html` (формула).

## P0 — блокеры

### FIX-01 · Единый источник mock-данных (главная = каталог = профиль)
```
Создай единый модуль mock-данных участников (например lib/participants.ts) и используй его как ЕДИНСТВЕННЫЙ источник для лендинга (блок «Последние записи»), каталога и страниц профиля /p/{id}.

Требования:
- Один массив из 12 участников с полями: id (p-001…p-012), name, role, jurisdiction, domain, score (одна точность — десятые, напр. 94.7), tier (Platinum/Gold/Silver/Bronze/Watch), dominantFactor (код+название или "Сбалансирован"), status (active/provisional/watch), kybLevel, isVerifiedLegal, independentTests, updatedAt.
- Блок «Последние записи в реестре» на главной должен брать первые 7 участников ИЗ ЭТОГО ЖЕ массива. Имя, score, тир и ссылка /p/{id} обязаны совпадать с тем, что открывается на самой странице профиля.
- Сейчас баг: на главной "AlphaPep Lab" (p-001) ведёт на профиль "BioSynth Laboratories AG", а p-006/p-007 дают «Участник не найден». После фикса любой клик с главной открывает того же участника с теми же числами.
- Точность score единая во всех местах (десятые: 94.7, 81.0, …). Тир вычисляется по порогам Platinum 88–100 / Gold 72–87 / Silver 55–71 / Bronze 35–54 / Watch <35 (как на /tiers).
Не меняй визуальный стиль карточек.
```

### FIX-02 · Засеять «Лог инцидентов» и связать с флагами/Watch
```
Страница /incidents сейчас пустая, но на главной заявлена как ключевая ценность, и в каталоге есть участник тира Watch и флаги в профилях. Наполни лог инцидентов из единого mock-источника.

Требования:
- Добавь 3–4 mock-инцидента с полями: дата (timestamp), участник (ссылка на /p/{id}), тип нарушения, серьёзность (INFO/WARNING/CRITICAL), краткое описание, anchor-хэш (моноширинный, обрезанный), статус (open/upheld/dismissed).
- Сортировка по дате (новые сверху). Бейдж серьёзности цветом: CRITICAL — error #b3261e, WARNING — янтарный, INFO — нейтральный.
- Участник тира Watch и любой участник с флагом в профиле (вкладка «Обзор» → «Флаги») должны иметь соответствующую запись в /incidents; в профиле флаг даёт ссылку «Открыть в логе инцидентов →».
- Сохрани блок про принцип неизменяемости и кнопку «Сообщить о нарушении →».
Пустое состояние оставь как fallback на случай отсутствия данных.
```

```
Сейчас в хедере одновременно «Войти» и «Кабинет», а /dashboard и /submit открыты без логина. Введи mock-состояние авторизации.

Требования:
- Добавь mock auth (например через cookie/localStorage флаг + контекст). Дефолт — НЕ залогинен.

- /dashboard и /submit: если НЕ залогинен — редирект на /login?next=… или показ экрана «Войдите, чтобы открыть кабинет» с кнопкой входа. После mock-логина — доступ.
- Кнопка «Войти» на /login по сабмиту ставит флаг и ведёт на /dashboard.
Не трогай визуальные токены.
```

## P1 — целостность флоу и соответствие спекам

### FIX-04 · Починить /forgot-password
```
Страница /login ссылается на /forgot-password, который отдаёт 404. Создай /forgot-password.

Требования:
- Карточка по центру в стиле /login: заголовок «Восстановление доступа», поле Email, кнопка «Отправить ссылку», ссылка «← Вернуться ко входу» (/login).
- Состояние успеха: «Если аккаунт существует, мы отправили ссылку на {email}».
- Тот же визуальный стиль, что и /login (карточка, отступы, токены).
```

### FIX-05 · Выбор «в сравнение» в каталоге → /compare
```
В пустом состоянии /compare сказано «выберите участников из каталога», но в карточках каталога нет действия добавления к сравнению. Замкни флоу.

Требования:
- В карточку каталога добавь чекбокс/кнопку «＋ В сравнение» (не ломая клик по карточке → профиль).
- Выбранные ID копятся; внизу/сверху появляется липкая панель «Сравнить (N)» с переходом на /compare?ids=a,b,c (2–5 участников; при >5 — дизейбл с подсказкой).
- /compare читает ids из query и строит таблицу сравнения по факторам; ограничение одной ролью с warning при несоответствии (как уже задумано в US-1.1).
- Сохрани существующее пустое состояние и «Добавить по ID».
```

### FIX-06 · Апелляция (/appeals/new) + чистка мёртвых ссылок кабинета
```
По wireframes есть экран апелляции (/appeals/new), сейчас /appeals = 404; в кабинете «Алерты контрагентов → Просмотреть» ведёт в никуда.

Требования:
- Создай /appeals/new: выбор оспариваемого флага (dropdown), поле «Заявление/доказательства» + загрузка файла, предупреждение про loser-pays, кнопка «Подать».
- Добавь статус-трекер апелляции: submitted → panel_forming (VRF) → in_review → decided (горизонтальные шаги, текущий подсвечен, aria-current).
- В профиле у флага и в кабинете в карточке «Алерты» кнопки ведут на /appeals/new?flag=… (или на список апелляций, если решишь сделать /appeals как список).
- Если страница помечена как V1 — оформи доступной заглушкой «Скоро (V1)», но без 404.
Стиль — текущие токены.
```

### FIX-07 · Скелетоны и состояния ошибки вместо «Загрузка…»
```
Сейчас /dashboard (и др. клиентские страницы) показывают голый текст «Загрузка...». По design-system нужны skeleton-состояния.

Требования:
- Замени все плейсхолдеры «Загрузка...» на skeleton-компоненты (shimmer: linear-gradient 90deg #eee/#f5f4f1/#eee, анимация 1.3s) для: каталога (скелетоны карточек), профиля (скелетон шапки+карточек), кабинета (скелетон Score-героя и карточек).
- Добавь error-состояние: «Не удалось загрузить · Повторить» с кнопкой retry и сохранением request_id (моноширинный), для каталога и профиля.
- Деградация внешних данных не должна ломать render страницы.
```

### FIX-08 · Убрать B2B из primary-nav + выровнять формулу Trust Score
```
Две правки прозрачности/навигации.

1) B2B сейчас в основном меню равно с Каталогом/Сравнением, но это заглушка «Скоро». Убери B2B из основной навигации хедера; оставь ссылку в футере («Для организаций (B2B)») и/или как вторичный CTA. В бургер-меню тоже убрать из основного списка.

2) На /trust-model формула показана как Σ(wk·Fk·Vi,k)/Σwk — без Dᵢ (свежесть/decay) и Pₛ (штрафы). Приведи к канонической из spec-algorithm: TS = Σ(Fᵢ·Wᵢ·Dᵢ·Vᵢ)·(1−Pₛ). Добавь краткие пояснения Dᵢ (decay по свежести) и Pₛ (штраф за подтверждённые нарушения) рядом с формулой.
```

## P2 — полировка, доступность, контент

### FIX-09 · Единая точность score
```
Приведи отображение Trust Score к одной точности во всём продукте — десятые (напр. 94.7, 81.0, 67.0). Сейчас главная показывает целые (94), а каталог/профиль — десятые (94.7). Используй tabular-nums везде.
```

### FIX-10 · Усилить форму /report
```
Форма /report не привязывает жалобу к участнику и не принимает доказательства.

Требования:
- Добавь поле «Участник (ID/домен)» с автоподсказкой из каталога (необязательное, но рекомендуемое).
- Добавь загрузку файла-доказательства (PDF/PNG/JPG, до 20 MB), как на /submit.
- Сохрани конфиденциальность и текст про 72 часа. Состояние успеха: «Сообщение принято, ID обращения {request_id}».
```

### FIX-11 · Доступность (a11y)
```
Точечные правки доступности без смены визуала:
- Контраст вспомогательного текста: где используется #95a4ba на белом для важной информации — затемнить до уровня WCAG AA (≥4.5:1 для основного, ≥3:1 для крупного).
- На баре фактора в профиле всегда показывать числовое значение текстом рядом (не только длина/цвет).
- Степпер в /claim и /claim/kyb: завершённые шаги помечать как «выполнено» (галочка + aria-label), текущий — aria-current="step".
- Горизонтальная лента табов профиля (Обзор/Факторы/История/Проверка целостности/Связи): добавить аффорданс прокрутки (градиент-затухание справа) и поддержку клавиатуры.
```

### FIX-12 · Состояния каталога: «нет результатов» и provisional-карточка
```
По design-system у каталога должны быть пустое и provisional состояния.

Требования:
- Empty-состояние фильтров: когда под фильтры нет участников — «Ничего не найдено — измените фильтры» + кнопка «Сбросить фильтры».
- Карточка provisional/New: вместо числа score — бейдж «New / Provisional» (tier-new #5b78a8) с подписью «накопление данных» (как в design-system tbadge tier-new). Watch-карточка — со score и красным акцентом.
- Кнопка «Показать ещё (N)» — курсорная подгрузка, сохранить.
```
