API Conventions, Errors & Data Dictionary

Единые правила для всех эндпоинтов: формат ошибок, пагинация, идемпотентность, версионирование, плюс словарь данных с семантикой ключевых полей и enum-значений.

02 — Соглашения
ТемаПравило
Базовый путь/v1; именование ресурсов во множественном числе (/participants)
ФорматыJSON (UTF-8); даты — ISO-8601 UTC; id — UUID
ПагинацияКурсорная: cursor + limit (≤100); ответ {data[], next_cursor}
Сортировка/фильтрыQuery-параметры; tie-break ранжирования: Vᵢ-покрытие → свежесть → хеш (BR-RANK)
ИдемпотентностьВсе write — заголовок Idempotency-Key; повтор с тем же ключом возвращает исходный результат
КонкурентностьОптимистичная блокировка / ETag для профиля; конфликт → 409
Аутентификацияread — public; write — X-Api-Key + X-Signature (PKI)
Rate limitPer-key/per-IP; превышение → 429 + Retry-After
ТрассировкаКаждый ответ содержит request_id (в т.ч. в ошибках)
03 — Единый формат ошибки
{
  "code": "RULE_VIOLATION",          // машиночитаемый код
  "message": "Custody incomplete; Vi capped",
  "details": { "rule": "BR-CUSTODY-SIG", "field": "lot_qr_key" },
  "request_id": "req_a1b2c3"
}

HTTP-статус задаёт класс, code — конкретную причину. Клиенты ветвятся по code, не по тексту.

Источник истины

Полные нормализации факторов — в Wave 1; бизнес-правила — в Business Rules; физические типы — в DB Schema.

06 — Версионирование и депрекация
  • Major в пути (/v1, /v2) — несовместимые изменения.
  • Аддитивные изменения (новые поля/эндпоинты) — без смены версии; клиенты игнорируют незнакомые поля.
  • Депрекация: заголовок Deprecation + Sunset, период поддержки, запись в changelog.
  • algo_version в ответах — версия параметров скоринга, ортогональна версии API.
