02 — Стиль архитектуры

Решение: модульный монолит first

Решение (см. ADR-001)

MVP/V1 — модульный монолит (modular monolith) с чёткими границами модулей и выделением только двух процессов вне ядра: асинхронный worker (batch/события) и signing-сервис (изоляция ключей). Микросервисы — отложенная эволюция при доказанной необходимости (нагрузка/команда), а не стартовая позиция.

КритерийМонолит-firstМикросервисы
Скорость разработки (lean-команда)✓ высокая✗ накладные расходы
Операционная сложность✓ низкая✗ оркестрация, сеть, observability
Транзакционная целостность скоринга✓ единая БД, ACID✗ распределённые транзакции
Независимое масштабирование частей◐ через реплики + воркеры✓ по сервисам
Риск «переписать через полгода»✓ модульность даёт швы для выноса✗ преждевременная сложность

Монолит делается модульным намеренно: границы модулей = будущие швы для выделения в сервисы. Это даёт преимущества монолита сейчас и опцион на микросервисы позже без переписывания.

Правило границ

Модули общаются через явные интерфейсы (не лезут в чужие таблицы напрямую). Это сохраняет возможность вынести любой модуль в отдельный сервис без переписывания вызывающих. Кандидаты №1 на вынос при росте: badge/rendering, scoring-worker, b2b.

06 — Ключевые потоки

Как данные проходят систему

Чтение профиля (горячий путь)

Покупатель → CDN (hit?) → SSR/BFF → catalog (read, кэш Redis) → ответ // p95 < 300мс; при miss — чтение из PostgreSQL реплики

Подача независимого теста

Оракул → API (protected, PKI) → ingestion (валидация, lot-QR, custody) → событие в очередь → Worker: scoring пересчёт (сразу) → anchoring (L2) → notifications (алерты затронутым)

Batch decay (раз в сутки)

Cron → Worker → scoring: применить Dᵢ ко всем профилям → пересчёт пропагации → пакетный anchoring (Merkle) → обновить кэш каталога

08 — Масштабирование
  • Read-путь: stateless SSR/BFF за CDN + read-реплики PostgreSQL + Redis-кэш. Масштабируется горизонтально.
  • Badge: кэшируемые подписанные изображения с TTL; CDN поглощает основную нагрузку.
  • Write/пересчёт: очередь + воркеры; пиковые события не блокируют read.
  • Batch: decay-пересчёт масштабируется по числу профилей (шардинг по диапазону id при росте).
  • Anchoring: Merkle-батчинг снижает число и стоимость L2-транзакций.
09 — Путь эволюции

Когда и что выносить в сервисы

ТриггерДействие
Нагрузка на бейдж >> остальногоВынести badge/signing в отдельный сервис (уже изолирован)
Тяжёлый скоринг тормозит APIВынести scoring-worker в отдельный сервис со своей репликой данных
B2B растёт независимоВынести b2b-модуль (API/вебхуки/аналитика) в сервис
Команда выросла (>2 команды)Резать по границам модулей §05, по одному, с контрактами
Принцип эволюции

Выделение сервисов — реакция на доказанную боль (нагрузка/организация), а не упреждающее усложнение. Модульные границы и контракты, заложенные сейчас, делают выделение дешёвым тогда, когда оно действительно понадобится.
