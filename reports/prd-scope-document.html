Scope Document — границы проекта

Якорь против scope creep. Фиксирует, что входит и не входит в продукт в целом и по фазам (MVP / V1 / V2). Любое расширение проходит через процесс изменения scope (§07). Развивает границы из Vision §08 и Roadmap.

Главный принцип scope

PeptideTrust — слой доверия, а не маркетплейс, не эскроу и не медицинский сервис. Всё, что приближает продукт к торговле товаром, удержанию средств или медицинским рекомендациям — вне scope по определению, независимо от фазы.

02 — Глобальный In / Out

Что продукт делает и не делает (вне зависимости от фазы)

✓ В границах (всегда)

  • Открытый публичный реестр доверия и Trust Score
  • Агрегация и независимая верификация данных (оракулы, тесты, артефакты)
  • Прозрачный алгоритм Score с доминирующим фактором
  • Встраиваемый бейдж и публичное read-API
  • Идентичность (KYB/KYC), PKI, ончейн-якоря целостности
  • Апелляции и управление (Steward-панели)
  • B2B-продукты для «решающих» сторон

✗ Вне границ (всегда)

  • Маркетплейс / продажа товара / листинг цен на продажу
  • Эскроу, удержание средств, процессинг платежей за сделки
  • Медицинские рекомендации, дозировки, протоколы применения
  • Гарантия/страхование качества товара
  • Продажа влияния на Score оцениваемой стороне
  • Анонимная агрегация без верификации (мы не «ещё один форум»)
03 — Scope MVP

MVP Доказать ценность для спроса (pull)

✓ В MVP

  • Публичный каталог + профили + Score с доминирующим фактором
  • 5 якорных ролей (полностью оцениваются): MANUFACTURER (завод-синтез / API-producer, upstream B2B), VENDOR (RUO-вендор — витрина/бренд), LABORATORY, DISTRIBUTOR, RETAILER (buyer-facing, но потолок тира Gold + «источник не верифицирован» до Vᵢ-пропагации V1; см. ADR docs/adr-role-taxonomy.md)
  • Сегментация по ролям, фильтры, сравнение в сегменте
  • Встраиваемый подписанный SVG/PNG бейдж + аналитика кликов
  • Публичное read-API (профиль, история, верификация anchor_hash)
  • Claim-profile + KYB/KYC (L0–L2) + waitlist-онбординг
  • New/Provisional fast-track (KYB + 1 blind-тест)
  • Независимые тесты (оракулы) + базовый Vᵢ + COA с PKI
  • Ончейн-якоря целостности (L2)
  • Unclaimed-профили (нейтральные факты, политика P0)
  • Базовые Pₛ-флаги и decay (Dᵢ)

✗ Не в MVP (позже)

  • Trust-пропагация Vᵢ между контрагентами → V1
  • Chain Pull-механика → V1
  • Полный апелляционный процесс + Steward-панели → V1
  • B2B-SKU (due-diligence, мониторинг) → V1
  • Lot-QR chain-of-custody (полная) → V1
  • ZK/shielded assertion → V2
  • Локализация (не-английский) → V2
MVP exit criteria (из Волны 3)

Переход к V1 — по достижению KPI фазы MVP (рост North Star «проверенные решения», доля профилей с ≥1 независимым подтверждением, активные податели данных), а не по календарю.

04 — Scope V1

V1 Запустить маховик доверия и первую выручку

✓ В V1

  • Trust-пропагация Vᵢ (одношаговая, λ_blind, агрегация, потолок)
  • Алерты по контрагентам + демпфер с отменой буфера
  • Chain Pull (втягивание контрагентов, кредиты)
  • Апелляции + Steward-панели (VRF-отбор, conflict-screening, chair-оракул)
  • B2B-SKU: due-diligence отчёты + мониторинг-API/вебхуки
  • Lot-QR chain-of-custody + анти-reuse артефактов
  • Расширенная анти-gaming (collusion-detector, cyclic-ban)
  • Transparent Affiliate, Verified Curator, Verified-Legal сегмент

✗ Не в V1

  • Полный ZK (используется shielded assertion) → V2
  • Аналитика-SKU и VaaS → V2
  • Расширение на новые категории/регионы → V2
  • Опциональный инфра-модуль процессингов → V2
05 — Scope V2

V2 Устойчивость, охват, продвинутая приватность

✓ В V2

  • Полноценный ZK-раскрытие (замена shielded assertion)
  • Аналитика-SKU (агрегированные тренды) + VaaS (интеграция в чужие платформы)
  • Расширение категорий/регионов; локализация (i18n)
  • Опциональный модуль инфра-надёжности процессингов
  • Оптимизация юнит-экономики, снижение CAC

✗ По-прежнему вне границ

  • Маркетплейс / эскроу / платежи за сделки
  • Медицина / дозировки
  • Платная позиция/влияние на Score
06 — Явные исключения с обоснованием

Почему этого нет — чтобы не возвращаться к спорам

ИсключеноПочему
Эскроу / удержание средствЮридический и операционный риск (финрегулирование); прямое решение пользователя — доверие держится на репутации GB-организаторов, не на эскроу
Маркетплейс / продажиНарушает нейтралитет (мы бы стали стороной сделки); смешивает trust-layer с торговлей
Медицинские рекомендацииRUO-позиционирование; регуляторный риск; вне компетенции
Платная позиция в каталоге / влияние на ScoreУничтожает доверие (конфликт интересов как у платных рейтингов); платит «решающий», не оцениваемый
Полный Trust Score для неформальных продавцовНет верифицируемых данных; остаются как репутационная заметка
Процессинги в core-ScoreНет механизма оценки в рамках доверия к товару; вынесены в опциональный инфра-модуль
07 — Процесс изменения scope

Как scope может меняться

  1. Предложение: любое расширение оформляется как запрос с обоснованием ценности и фазой.
  2. Проверка границ: не нарушает ли «всегда вне границ» (§02, §06). Если нарушает — отклоняется без обсуждения деталей.
  3. Влияние: оценка на North Star/KPI, юнит-экономику, безопасность, нейтралитет.
  4. Решение и версия: принятое изменение фиксируется с обновлением версии Scope Document и, при необходимости, algo_version/Roadmap.
  5. Реестр: отклонённые/отложенные — в backlog с пометкой фазы.
Зачем формальный процесс

Без явного фильтра «всегда вне границ» проект расползётся под давлением частных запросов ролей (это видно из оценки 22 ролей рынка — у каждой роли свои хотелки). Процесс защищает нейтралитет и фокус MVP.
