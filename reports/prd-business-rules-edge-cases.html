Каталог бизнес-правил и edge cases

Единый справочник правил, лимитов, валидаций и нестандартных ситуаций — снимает большую часть вопросов разработчиков. ID правил (BR-*) используются в User Stories и PRD. Консолидирует решения v2-реестра, Волн 0–4 и закрытий P0/P1/P2 — новых решений не вводит.

Числовые значения помечены как провизорные (v0.9) и проходят SME-валидацию (см. P0-closure → v1.0). Здесь зафиксированы правила и их структура.

01 — Правила Score
Формула и нормализация
IDПравило
BR-SCORE-1Итоговый Score: TS = Σ(Fᵢ·Wᵢ·Dᵢ·Vᵢ)·(1−Pₛ)
BR-SCORE-2Сумма весов роли строго ΣWᵢ = 1.0; веса задаются матрицей 22 роли × 9 факторов
BR-SCORE-3Fᵢ нормируется по абсолютным бандам (0–100), не относительно конкурентов
BR-SCORE-4Отсутствующие данные по фактору → Fᵢ трактуется как 0 (нет данных ≠ хорошо)
BR-SCORE-5Fᵢ из источника — отделён от надёжности источника (источник влияет через Vᵢ, не через Fᵢ)
BR-RANKТай-брейк при равном Score: покрытие Vᵢ (доказанность) → свежесть данных → детерминированный hash(id‖epoch)
BR-DOMFACTORДоминирующий фактор = max(Fᵢ·Wᵢ·Dᵢ·Vᵢ); при отсутствии явного лидера (разрыв ниже порога) — «сбалансированный профиль»; роли без полного TS — репутационная заметка
BR-DISPLAYScore всегда отображается с подписью доминирующего фактора (каталог, профиль, бейдж, API); для provisional/без полного TS вместо числа — плашка статуса/репутационная заметка
02 — Правила Vᵢ и пропагации
IDПравило
BR-VI-1Базовый Vᵢ: 0.40 + (Sub_counterparty/100)·0.55
BR-VI-2Независимый тест (custody полна) → Vᵢ=1.0; vendor-sample → Vᵢ≤0.70; безартефактный отзыв → Vᵢ≈0.45
BR-VI-MULTIПри нескольких независимых источниках по одному фактору берётся max(V_source) (сильнейшее независимое подтверждение)
BR-VI-CONFLICTКонфликтующие источники (противоречие) → фактор замораживается, запускается разбор; не усредняется
BR-PROPПропагация одношаговая, переносит профильный субскор (не весь TS); S_prop — взвешенное по lot-share среднее; λ_blind=0.85 для blind; потолок V_prop ≤ 0.85
BR-PROP-CONCКонцентрация источников top_share > 0.60 → штраф за концентрацию (диверсификация поощряется)
BR-PROP-CYCLEЦиклические связи исключаются из пропагации (запрет для cyclic-links)
BR-PROP-BLENDИтоговый Vᵢ — смесь базового и пропагированного по покрытию (coverage-weighted)
BR-CHAINPULLСубскор контрагента влияет на Vᵢ участника (раздел 03 v2); приглашение/улучшение контрагента («Chain Pull») повышает собственный Vᵢ — давление вверх по цепочке
03 — Правила Dᵢ (decay)
IDПравило
BR-DI-1Экспоненциальное затухание по классу данных; 4 класса: A (долгоживущие, T½=365д), B (качество, 120д), C (логистика партии, 45д), D (поведение, 90д)
BR-DI-2Каждый класс имеет floor (Dᵢ не падает ниже минимума, чтобы старые подтверждения не обнулялись полностью)
BR-DI-3Decay пересчитывается батчем раз в сутки (BR-CADENCE-2)
04 — Правила Pₛ и демпфера
IDПравило
BR-PS-1Штрафы агрегируются: Pₛ = 1 − ∏(1−pₖ)
BR-PS-2Жёсткие потолки тиров: серьёзные нарушения ограничивают верхний тир независимо от прочих факторов
BR-PS-SWAPПодтверждённый sample-swap → отдельный pₖ на вендора (не на лабораторию)
BR-DAMPERВлияние падения контрагента сглаживается 30-дневным средним + алерт
BR-DAMPER-CANCELЕсли после алерта участник продолжает новое сорсинг у flagged-контрагента → 30-дневный буфер отменяется, пересчёт вниз сразу
BR-DAMPER-SEVУровни алерта: INFO / WARNING / CRITICAL — определяют срочность и видимость реакции
BR-CADENCE-1Критичные события (Pₛ-флаг, новый тест, отмена демпфера) → пересчёт сразу + анкоринг
BR-CADENCE-2Decay и пропагация → батч раз в сутки
05 — Идентичность и статусы
IDПравило
BR-KYB-LEVELSУровни L0 (PoP) → L1 (KYC) → L2 (KYB бизнес) → L3 (лицензии/расширенный)
BR-POPL0: уникальность телефона + device/CAPTCHA; опционально сторонний PoP-апгрейд
BR-KYB-FALLBACKЮрисдикция без KYB → fallback L1 + повышенный вес blind-тестов + потолок доверия (не выше заданного тира) до полного KYB
BR-PROVISIONALNew/Provisional → полный TS по доказательствам: KYB + ≥N независимых подтверждений по ключевым факторам + нет активных CRITICAL-флагов (не по календарю)
BR-UNCLAIMEDUnclaimed-профиль: только нейтральные факты из легитимных публичных источников; без негатива/Score до claim; механизм возражения/удаления; явная пометка
BR-NEUTRALITYОцениваемая сторона не может платить за влияние на Score/позицию/флаги; платит только «решающий»
BR-VERLEGALVerified-Legal (лицензированные каналы) отделяются фильтром сегментации
BR-PRIVACY-ZKShielded-уровень: публичность проверки без публичности личности; для посредников blind-связь доказывает «источник верифицирован, субскор ≥ X» без раскрытия имени; полный ZK — V2
BR-ANALYTICSB2B-аналитика продаётся только агрегированной; продажа влияния на конкретный Score/позицию/флаги запрещена (следствие BR-NEUTRALITY); V1+/V2
06 — Custody и артефакты
IDПравило
BR-CUSTODYНезависимый сэмплер (Vᵢ=1.0): покупатель / GB-организатор / аккредитованный аудитор; самозабор вендора никогда не =1.0
BR-CUSTODY-SIGПодпись на каждом шаге custody; разрыв/отсутствие подписи → тест не получает 1.0 (трактуется как vendor-grade 0.70)
BR-LOTQRCOA/артефакты привязываются к партии через lot-QR; «общий» сертификат без привязки не засчитывается на уровне партии
BR-ARTIFACT-REUSEАнти-reuse: повторно поданный медиа-артефакт отклоняется детектом
BR-COA-PKICOA принимается только с валидной PKI-подписью аккредитованного оракула; при отзыве ключа (CRL) — недействителен
07 — Lifecycle-правила
IDПравило
BR-ROLE-CHANGEСмена роли: Score не переносится напрямую (разные Wᵢ); общие факторы переносятся с сохранением Dᵢ; новая роль стартует Provisional
BR-MERGEОбъединение профилей при доказанной тождественности: сохраняется худший из активных Pₛ-флагов (негатив не смывается)
BR-SPLITРазделение: новые сущности — Provisional; подтверждённые флаги наследует правопреемник (нет ухода от ответственности дроблением)
BR-RECALLОтзыв партии/COA: lot → recalled, связанные подтверждения теряют вес, пересчёт сразу; повторные recall повышают Pₛ; публичный лог
BR-OFFBOARDОффбординг: tombstone — исторический Score и подтверждённые Pₛ-флаги публичны, ПДн минимизируются, ончейн-хеши сохраняются
BR-STEWARD-LCSteward/оракул: фиксированный срок + ротация + снятие по триггерам (конфликт, нарушение, неактивность); прозрачный лог
BR-RETENTIONПодтверждённые Pₛ-флаги мошенничества удерживаются на срок наказания (защита третьих лиц), затем tombstone
BR-APPEAL-PANELАпелляционная панель формируется VRF-отбором Steward + conflict-screening + chair-oracle (кросс-сегментная, v2 §07)
BR-APPEAL-BINDINGРешение апелляции binding, публикуется в реестре; правок «задним числом» нет (стандарт неизменности)
BR-APPEAL-COSTLoser-pays: проигравшая сторона несёт издержки разбора; предупреждение при подаче
BR-APPEAL-ESCALATIONЭскалация по уровням L1→L3 в зависимости от сложности/значимости спора; не лишает права на суд
08 — Лимиты и валидации
IDПравило
BR-LIM-RATERate-limit по API-ключу на write/protected; публичный read кэшируется и rate-limited по IP
BR-LIM-WRITE-AUTHWrite-эндпоинты: обязательны API-key + payload-подпись; иначе отклонение
BR-VAL-INPUTВсе входные данные (отзыв, тест, COA, claim) валидируются по схеме; невалидное — отклоняется с ошибкой
BR-VAL-DUPДубликаты подтверждений от одного источника не суммируются (защита от накрутки покрытия)
BR-VAL-INDEP«Независимость» источников проверяется графом связей; зависимые источники не считаются за независимые подтверждения
BR-BADGE-CACHEБейдж кэшируется с TTL; значение синхронизируется с последним пересчётом; подпись обязательна
BR-VERSIONКаждый Score-результат привязан к algo_version; новые параметры применяются forward-only; ретро — только для errata
BR-APIВерсионирование пути (/v1); публичный read кэшируется и rate-limited по IP, protected write требует API-key + payload-подпись; /verify/{anchor_hash} публичен (контур — OpenAPI)
BR-ANCHORЦелостность через ончейн: анонимный солёный хеш score-событий/COA/custody, Merkle-батчинг в L2; payload без ПДн/идентификаторов; проверка через /verify
