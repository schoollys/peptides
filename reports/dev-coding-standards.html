Coding Standards & Repository Structure

Конвенции, чтобы код был однородным, а границы модульного монолита (ADR-001) — соблюдались. Стек TypeScript/Node (ADR-009).

03 — Границы модулей (модульный монолит)
  • Модуль экспортирует публичный интерфейс (сервис/порты); внутренности не импортируются напрямую.
  • Межмодульное взаимодействие — через интерфейсы или доменные события (event_outbox), не через прямой доступ к чужим таблицам.
  • У каждого модуля — своя зона данных; общие типы — в packages/shared.
  • Зависимости направлены к ядру (scoring), а не наоборот; цикл-импорты запрещены линтером.
  • Это готовит безболезненный вынос модуля в сервис (эволюция, ADR-001).
Почему важно

Чёткие границы — главная страховка модульного монолита от превращения в «большой ком грязи». Нарушение границы ловится на CI (правила импорта).

ЯзыкTypeScript strict; без any без обоснования
ФорматированиеPrettier (автомат); ESLint с правилами границ модулей
ИменованиеcamelCase (переменные/функции), PascalCase (типы), CONSTANT_CASE (конфиг)
Чистые функцииСкоринг-математика — чистые, детерминированные (тестируемость/регрессия)
Без хардкодаВсе tunables — из Parameters Registry по algo_version/конфигу
КомментарииОбъясняют «почему», не «что»; ссылка на BR-*/ADR при неочевидной логике

06 — Ошибки, логи, конфиг
  • Ошибки API — единый формат {code,message,details,request_id} и каталог кодов (API Conventions).
  • Логи структурированные (JSON), с request_id/trace_id; без ПДн (см. Observability).
  • Конфиг — через env + типизированный загрузчик; секреты — только из секрет-менеджера.
  • Идемпотентность write — обязательна (Idempotency-Key); конкурентность — оптимистичная блокировка/ETag.
