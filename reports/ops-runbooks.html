Runbooks / On-call Playbooks

Пошаговые процедуры для дежурного на инциденты — операционные и security. Каждый runbook: симптом → детект → шаги → эскалация → восстановление → постмортем. Опирается на Волну 4 (incident response, SLA, threat model), Integration Contracts (обработка сбоев) и Infra & DevOps (бэкапы/DR).

РольЗона ответственности
On-call инженерПервичный детект, триаж, выполнение runbook, эскалация
Incident CommanderКоординация S1/S2, коммуникация, решение о публичном статусе
Security leadSecurity-инциденты (RB-07…RB-10), ротация/ревокация ключей
DPO / LegalУтечка ПДн (RB-09), нотификации регулятору
Параметрический комитетРешения по алгоритму/весам (RB-11, RB-12)
01 — Severity и эскалация
УровеньКритерийРеакция / эскалация
S1Недоступен публичный read-путь; утечка ПДн; компрометация ключа подписантаНемедленно: IC + security lead; публичный статус
S2Деградация (anchoring/write/вебхуки стоят); сбой одной интеграцииOn-call → IC при затягивании; запись в журнал
S3/S4Локальные сбои, бэклоги без влияния на read; минорныеOn-call в рабочем порядке
Инвариант

Сбой любой внешней интеграции (KYB/L2/KMS/вебхуки) НЕ должен ронять публичный read-путь. Если роняет — это автоматически S1.

RB-01KYB/KYC-провайдер недоступенS2
Симптом
Claim/верификации зависают; рост pending_verification; ошибки UPSTREAM_UNAVAILABLE.
Детект
Алерт по error-rate провайдера и длине очереди верификаций.
Шаги
  1. Подтвердить, что read-путь и Score не затронуты (только онбординг).
  2. Проверить статус-страницу провайдера; включить fallback-провайдер по юрисдикции (Wave 2).
  3. Убедиться, что заявки остаются в очереди (не теряются), ретраи идут.
  4. Опубликовать баннер «верификация замедлена» при затягивании >SLA.
Восстановление
После восстановления — дренаж очереди; проверить отсутствие дублей claim (идемпотентность).
RB-02Сбой L2 RPC / anchoringS2
Симптом
Рост score_events.anchor_status='pending'/'failed'; ошибки RPC.
Детект
Алерт по глубине очереди anchoring и возрасту самого старого pending.
Шаги
  1. Подтвердить: Score валиден off-chain, целостность догоняется — read не затронут.
  2. Проверить здоровье RPC-эндпоинта; переключить на резервный provider/endpoint.
  3. Убедиться, что ретраи с backoff активны (SM anchoring); не сбрасывать batchId (идемпотентность).
  4. При длительном сбое сети — переключение Base↔Arbitrum через HashAnchor-абстракцию (ADR-004), решение IC.
Восстановление
Дренаж pending-батчей; выборочная проверка /verify/{hash}.
RB-03KMS/HSM недоступенS1→S2
Симптом
Не подписываются бейджи/лоты/якоря; ошибки signing-сервиса.
Детект
Алерт signing-сервиса и роста ошибок подписи.
Шаги
  1. Бейджи отдаются из кэша (TTL/ETag) — подтвердить, что публичный показ жив.
  2. Новые подписи/anchoring ставятся в очередь, не теряются.
  3. Проверить доступность KMS/HSM-региона; failover на резервный регион/реплику.
  4. Если недоступность затрагивает read дольше TTL → эскалация до S1.
Восстановление
Дренаж очереди подписей; ротация при подозрении на компрометацию → RB-08.
RB-04Сбой БД / восстановлениеS1
Симптом
Ошибки чтения/записи, недоступность основного инстанса.
Детект
Health-check БД, репликационный лаг, рост 5xx.
Шаги
  1. Failover на реплику (read-path в приоритете).
  2. При повреждении — restore из PITR к точке до инцидента (RPO/RTO цели, Infra & DevOps).
  3. После restore — реконсиляция event_outbox и anchoring (что не доехало).
  4. Проверить инварианты скоринга на свежем срезе (golden-датасет).
Восстановление
Подтвердить целостность; зафиксировать фактический RPO/RTO в постмортеме.
RB-05Бейдж/CDN перегрузка или DDoSS1
Симптом
Всплеск трафика на /badge и read; рост латентности/5xx.
Детект
Алерт по RPS/латентности; WAF-сигналы.
Шаги
  1. Убедиться, что бейдж/read за CDN с кэшем — край сети поглощает пик.
  2. Поднять TTL кэша бейджа; включить усиленные WAF-правила/rate-limit на источники.
  3. Защитить write/anchoring приоритетными лимитами; оракулам — приоритет.
  4. При таргетированном DDoS — провайдер anti-DDoS + блок-листы; IC решает о статусе.
Восстановление
Вернуть TTL/лимиты к норме; анализ источников.
RB-06Бэклог доставки вебхуков (B2B)S3
Симптом
Рост event_outbox.status='failed'/attempts; жалобы B2B-клиентов.
Детект
Алерт по длине очереди и доле failed.
Шаги
  1. Определить, общий ли это сбой воркера или конкретный недоступный endpoint подписчика.
  2. Endpoint подписчика мёртв → авто-пауза подписки + уведомление владельца (Integration Contracts §03).
  3. Сбой воркера → перезапуск; ретраи по backoff; ручной replay по delivery_id.
Восстановление
Дренаж очереди; подтвердить дедупликацию у получателей (at-least-once).
RB-07Компрометация ключа оракулаS1
Симптом
Подозрительные COA/подписи; сигнал от оракула об утечке.
Детект
Аномалии в подачах оракула; внешний репорт.
Шаги
  1. Немедленная ревокация сертификата через CRL (Wave 2 §04).
  2. Определить «дату компрометации vs дату подписи»: подписи ДО компрометации остаются валидными, ПОСЛЕ — аннулируются.
  3. Пометить затронутые factor_inputs/coas на пересчёт; снять их вклад.
  4. Перевыпуск ключа оракулу после расследования; уведомить затронутых участников.
Восстановление
Пересчёт Score затронутых; запись в governance_log + публичный инцидент-лог.
RB-08Компрометация ключа подписанта реестраS1
Симптом
Поддельные бейджи/контр-подписи, проходящие проверку.
Детект
Аномалии подписей; внешний репорт; аудит KMS-доступа.
Шаги
  1. Отозвать скомпрометированный ключ; задействовать резервный из KMS/HSM (промежуточный, корень офлайн).
  2. Перевыпустить подписи бейджей (серверный рендер — массовая перегенерация по новому ключу).
  3. Опубликовать новый публичный ключ верификации; старый — в CRL.
  4. Аудит: что подписано в окне компрометации.
Восстановление
Подтвердить, что валидны только новые подписи; постмортем + публичный лог.
RB-09Утечка ПДнS1
Симптом
Доступ/эксфильтрация данных из защищённого контура идентичности.
Детект
Аудит-лог доступа, DLP-сигналы, внешний репорт.
Шаги
  1. Изолировать контур; отозвать скомпрометированные доступы/токены.
  2. Подключить DPO/Legal: оценка объёма, нотификация регулятору в установленные сроки (GDPR), уведомление субъектов при необходимости.
  3. Зафиксировать таймлайн и затронутые данные (минимизация off-chain ограничивает объём).
  4. Закрыть вектор; ротация ключей шифрования при необходимости.
Восстановление
Постмортем; публичное раскрытие по политике; меры предотвращения.
RB-10Входящий responsible disclosureS2
Симптом
Сообщение исследователя об уязвимости через disclosure-канал.
Шаги
  1. Подтвердить получение; завести приватный тикет; присвоить severity.
  2. Воспроизвести; оценить эксплуатируемость и охват.
  3. Фикс в окне до публикации; согласовать раскрытие с репортёром.
  4. Опубликовать в инцидент-логе после фикса; (bug bounty — позже по мере роста).
RB-11Аномалия Score / баг алгоритмаS1
Симптом
Массовый необъяснимый сдвиг Score; расхождение с golden-датасетом.
Детект
Регрессия скоринга в CI; алерт по распределению Score; жалобы.
Шаги
  1. Зафиксировать algo_version и входы; воспроизвести на golden-датасете.
  2. Если баг в новой версии параметров → откат на предыдущую algo_version (forward-only: ввести корректирующую версию).
  3. Если баг в коде → rollback релиза (canary/auto-rollback, Infra & DevOps).
  4. Пересчёт затронутых; уведомление; запись в changelog/governance_log.
Эскалация
Изменение параметров/весов — только через параметрический комитет.
RB-12Кольцо сговора / массовый абьюзS2
Симптом
Граф-детектор: кластер аффилированных аккаунтов накручивает Vᵢ/отзывы.
Детект
Алерт collusion-ring detector; всплеск регистраций/отзывов.
Шаги
  1. Заморозить пропагацию по подозрительным циклам (BR-PROP-CYCLE).
  2. Сверить real_identity_id (скрытый) на аффилированность; пометить кластер.
  3. Снять вклад накрученных подтверждений; применить Pₛ при подтверждении.
  4. Спорные суждения — на кросс-сегментную Steward-панель (раздел 07 v2).
RB-13Массовый recall партийS2
Симптом
Множественный отзыв лотов/COA по одному вендору/оракулу.
Шаги
  1. Перевести лоты в recalled; связанные подтверждения теряют положительный вклад (BR-RECALL).
  2. Немедленный пересчёт (критическое событие, не batch).
  3. При системном источнике (оракул) — проверить на RB-07.
  4. Уведомить downstream-контрагентов (каскад через мониторинг-вебхуки).

Связка с тестами

Каждый постмортем порождает регрессионный тест/алерт, чтобы инцидент не повторился. Security-инциденты пополняют Security Test Cases.
