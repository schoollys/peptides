Детальные контракты интеграций: форматы вебхуков, retry-логика, обработка сбоев третьей стороны. Дополняет Integration Map (что/где) конкретными payload и политиками.

02 — Исходящие вебхуки (B2B-мониторинг)

Подписка через POST /b2b/webhooks. События: score.updated, flag.opened, lot.recalled.

POST {subscriber_url}
Headers:
  Content-Type: application/json
  X-PT-Event: score.updated
  X-PT-Delivery: 9f1c...           # уникальный id доставки (идемпотентность у получателя)
  X-PT-Signature: sha256=...       # HMAC payload ключом подписки
  X-PT-Timestamp: 1718800000

{
  "event": "score.updated",
  "delivery_id": "9f1c-...",
  "occurred_at": "2026-06-19T21:00:00Z",
  "subscription_id": "sub_...",
  "data": {
    "participant_id": "uuid",
    "score": 78.4,
    "previous_score": 82.0,
    "dominant_factor": "QEF",
    "algo_version": 1,
    "anchor_hash": "0x..."
  }
}
ПолеНазначение
X-PT-SignatureHMAC-SHA256 тела ключом подписки — получатель верифицирует подлинность
X-PT-DeliveryУникальный id — получатель дедуплицирует (at-least-once доставка)
X-PT-TimestampЗащита от replay (отклонять старые)
03 — Retry и гарантии доставки
ПараметрПолитика
СемантикаAt-least-once; получатель обязан дедуплицировать по delivery_id
УспехHTTP 2xx в течение таймаута (напр. 5с)
РетраиЭкспоненциальный backoff с джиттером: напр. 1м → 5м → 30м → 2ч → 6ч (до N попыток)
Хранениеevent_outbox (status, attempts, next_retry_at) — transactional outbox
ИсчерпаниеПосле N неудач — пометка failed, алерт владельцу подписки, ручной replay
ОтключениеДолго недоступный endpoint → авто-пауза подписки + уведомление
Transactional outbox

Событие пишется в event_outbox в одной транзакции с изменением данных — гарантия, что доставка не потеряется при сбое после коммита (паттерн из схемы БД).

04 — Входящие интеграции
ИсточникКонтракт
Оракул (COA)Подача через POST /submissions kind=coa с signed_payload (PKI); сервер верифицирует подпись по сертификату оракула (не отозван в CRL)
KYB/KYC провайдерAsync-результат верификации: callback/поллинг; мы сохраняем только статус/уровень + provider_ref
SMS/OTPЗапрос/проверка кода (PoP L0); rate-limit на отправку
Перевозчики (V1)Трекинг-события custody → ingestion (CCF); идемпотентность по tracking id
05 — Обработка сбоев третьей стороны
СценарийПоведение
KYB-провайдер недоступенClaim → состояние pending_verification; ретраи; read-путь не затронут; fallback-провайдер по юрисдикции
KMS/HSM недоступенБейджи — из кэша (TTL); новые подписи/anchoring ставятся в очередь; алерт on-call
L2 RPC сбойAnchoring остаётся pending, ретраи с backoff; Score валиден off-chain, целостность догоняется
Получатель вебхука падаетРетраи по политике §03; пауза подписки при длительном отказе
Платёжный провайдер (B2B)Изолирован; влияет только на биллинг, не на ядро/доверие

Общий инвариант: сбой любой внешней интеграции (кроме БД/кэша) не должен ронять критичный публичный read-путь. Запись/anchoring/доставка — асинхронны и устойчивы к сбоям через очереди и ретраи (Integration Map §05).
