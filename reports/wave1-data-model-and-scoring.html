01 — Разделение ролей Fᵢ / Vᵢ / Dᵢ

Чтобы не путать «насколько хорошо» с «насколько верим»

Ключевой принцип, который делает нормализацию чистой: каждая ось отвечает за своё и не дублирует другую.

МножительОтвечает на вопросИсточник
Fᵢ (0–100)Насколько хорош сам результат по фактору (чистота, удовлетворённость, достоверность)Содержание данных — эта спека
Vᵢ (0.40–1.00)Насколько можно верить источнику этого результата (независимость образца, тип оракула)Оракульная сеть (раздел 02 v2)
Dᵢ (0.10–1.00)Насколько результат свежийВозраст данных (раздел 03 v2)
Следствие для нормализации

Fᵢ кодирует только качество измеренного. Доверие к источнику в Fᵢ не подмешиваем — это делает Vᵢ. Поэтому, например, чистота 99% по тесту вендора и по blind-тесту дают одинаковый Fᵢ, но разный итоговый вклад из-за Vᵢ (0.70 против 1.00). Это убирает двойной учёт доверия.

⚠️ Приватность и удаление

Поля идентичности и upstream_ref в Disclosure хранятся off-chain с правом удаления. On-chain — только onchain_hash/onchain_tx из COA и ScoreEvent. При удалении связи участником: ребро исключается из расчёта (Vᵢ откатывается к базовым 0.40 по затронутым партиям), но хеши прошлых ScoreEvent остаются — аудит «что и когда считалось» сохраняется без раскрытия удалённого содержимого. Точную политику ретеншена доспецифицировать в PRD (§05/§09).

03 — Принципы нормализации Fᵢ

Общие правила для всех 9 факторов

1. Абсолютные банды

Каждый фактор задаётся кусочной функцией с фиксированными порогами (банды), а не перцентилями. Балл одного участника не зависит от других — это прозрачно и не геймится «утоплением» конкурентов.

2. Гибридная постура и «уверенность данных»

Если по фактору есть структурированные числа (например, чистота из COA) — считаем по бандам. Если есть только факт наличия артефакта без чисел (например, «COA приложен, но не распарсен») — даём частичный кредит с потолком до подтверждения. Важно: «уверенность в источнике» в основном несёт Vᵢ; потолок здесь — про неполноту самого содержания, а не про доверие.

Fᵢ = clamp(0, 100, band(raw_value)) // если есть число Fᵢ = min(band_floor_partial, presence_credit) // если только артефакт без чисел

3. Отсутствие данных = 0 (но не «плохо»)

Незаполненный фактор входит в формулу как 0 (согласовано с разделом 05 v2: пустой фактор — упущенный потенциал). Чтобы новички не выглядели как нарушители, действует статус New/Provisional (раздел 03 v2) — отдельная вкладка, а не низ выдачи.

4. Мульти-оракульный потолок (факторы с весом ≥20%)

Если фактор весит ≥20% в роли, но подтверждён лишь одним источником, его Fᵢ ограничивается потолком до появления второго независимого сигнала. Это формализует требование из раздела 06 v2.

if W(factor) ≥ 0.20 and independent_sources < 2: Fᵢ = min(Fᵢ, 60) // потолок до второго источника — под валидацию

«Независимый источник» = другой оракул или verified-сигнал, не связанный с первым (другая лаборатория, verified-покупатель, кросс-claim контрагента).

04 — QEF: Quality & Efficacy

Качество продукта из COA/теста

Самый «числовой» фактор: на входе — измеримые величины анализа. Identity — это жёсткий гейт (не тот пептид → качество = 0), остальное — взвешенная смесь.

Входные сигналы

  • Identity (MS/MS-подтверждение, что это заявленный пептид) — pass/fail гейт.
  • Purity (HPLC, %) — основной непрерывный сигнал.
  • Net peptide content / label accuracy (соответствие заявленной массе/дозе, %).
  • Impurities / endotoxin / sterility (для инъекционного применения) — штрафной/гейтовый компонент.

Банды чистоты (purity → балл) под валидацию

Чистота HPLCБалл компонента purity
≥ 99.0%100
98.0–98.99%90
95.0–97.99%75
90.0–94.99%55
80.0–89.99%30
< 80% или identity fail0 (+ триггер Pₛ, если заявлено иное)

Сборка QEF

if identity == fail: QEF = 0 // жёсткий гейт else: QEF = 0.60·purity_band + 0.25·label_accuracy_band + 0.15·safety_band // endotoxin/sterility/impurities QEF = QEF × multi_oracle_cap_factor // см. §03.4

Веса компонентов (0.60/0.25/0.15) и банды label_accuracy/safety — дефолты под валидацию. Для неинъекционных RUO safety_band может быть нерелевантен → его вес перераспределяется на purity.

Пример

Вендор: blind-тест показывает purity 98.4%, identity pass, label accuracy 97% (≈100 балл), endotoxin в норме (safety 90).

QEF = 0.60·90 + 0.25·100 + 0.15·90 = 54 + 25 + 13.5 = 92.5.

Источник — blind-тест (Vᵢ=1.0), данные свежие (Dᵢ=1.0). Если бы тот же результат был по образцу вендора — QEF тот же 92.5, но Vᵢ=0.70 снизит итоговый вклад.

05 — CVF: Customer Verification

Качество и объём верифицированных отзывов

Не «сколько звёзд», а «сколько проверяемых довольных покупателей и нет ли нерешённых жалоб». Объём даёт уверенность, но с насыщением — чтобы 1000 отзывов не были в 1000 раз весомее 10.

Входные сигналы

  • N_verified — число отзывов с подтверждённым артефактом покупки (verified-buyer).
  • satisfaction — средняя удовлетворённость по verified-отзывам (0–1).
  • verified_ratio — доля verified среди всех отзывов (защита от накрутки безартефактными).
  • unresolved_complaints — подтверждённые нерешённые жалобы (штраф).

Формула

confidence(N) = N / (N + k) // насыщение, k≈5 — под валидацию CVF_base = 100 · satisfaction · confidence(N_verified) CVF = CVF_base · (0.5 + 0.5·verified_ratio) // демпфер за безартефактные − complaint_penalty // напр. −12 за каждую нерешённую CVF = clamp(0, 100, CVF)

k (точка насыщения), коэффициент verified_ratio и величина complaint_penalty — дефолты под валидацию. Всплеск отзывов 3× замораживается анти-аномалийным детектором (раздел 06 v2) до проверки — frozen-отзывы в N не входят.

Пример

Вендор: 20 verified-отзывов, средняя удовлетворённость 0.9, verified_ratio 0.8, 1 нерешённая жалоба.

confidence = 20/25 = 0.8 → CVF_base = 100·0.9·0.8 = 72 → ×(0.5+0.5·0.8)=×0.9 → 64.8 → −12 = 52.8.

06 — TRF: Truthfulness / Claim Reliability

Достоверность заявлений и честность раскрытия

Измеряет, подтверждаются ли заявления участника (о заводе-источнике, цепочке, партнёрствах) и не ловили ли его на противоречиях. Здесь доминирует не число, а доля подтверждённого и история опровержений.

Входные сигналы

  • verified_claims / total_claims — доля кросс-подтверждённых заявлений.
  • completeness — полнота раскрытия на уровне партий (не «один Gold-источник, остальное скрыто»).
  • contradictions — заявления, опровергнутые контрагентом/оракулом (жёсткий удар).
  • nondisclosure_flags — нераскрытые платные связи (для Transparent Affiliate и т.п.).

Формула

TRF_base = 100 · (verified_claims / total_claims) · completeness TRF = TRF_base − 25·contradictions − 15·nondisclosure_flags TRF = clamp(0, 100, TRF) // подтверждённая ложь о происхождении → также Pₛ + потолок тира (раздел 03 v2)

Штрафы за противоречие (−25) и нераскрытие (−15) — дефолты под валидацию. Опровергнутое заявление — это не только удар по TRF, но и инцидент Pₛ; двойной учёт разведён: TRF отражает «насколько верить словам», Pₛ — наказание за доказанный обман.

Пример

Трейдер: 8 из 10 claim подтверждены, completeness 0.9, 0 противоречий, 0 нераскрытий.

TRF = 100·0.8·0.9 = 72. Появись одно опровергнутое заявление → 72 − 25 = 47 + отдельный инцидент Pₛ.

07 — Остальные 6 факторов (глубина MVP-3)

PCF, SCIF, FRF, CCF, CVF-B, RF — входы, формулы, банды, примеры

Доведены до того же уровня детализации, что QEF/CVF/TRF: входные сигналы → формула → пороги → пример. Все числовые банды и веса компонентов — инженерные дефолты под валидацию SME/Параметрического комитета; структура от конкретных чисел не зависит.

07.1 — PCF: Process & Compliance

Зрелость производственного/складского процесса: сертификации, валидация процесса, целостность упаковки. Доминирует у фасовки (CDMO), компаундинга, аптек, лабораторий, B2B-логистов.

Входные сигналы

  • cert_tier — высшая релевантная действующая сертификация (GMP/GDP/ISO) и кем выдана.
  • process_validation — задокументирована ли валидация процесса (асептика, лиофилизация, SOP).
  • integrity — tamper-evident упаковка и отсутствие инцидентов целостности.
  • recalls — отзывы партий / нарушения комплаенса в окне (штраф).

Банды сертификации (cert_band) под валидацию

Статусcert_band
Действующий GMP, аудит признанным органом100
Действующий GMP, второй-стороной / self-audit75
GDP / ISO 9001 действующий60
SOP задокументированы, без сторонней сертификации35
Нет / просрочено0

Сборка PCF

PCF = 0.45·cert_band + 0.30·process_band + 0.25·integrity_band − 20·recalls_in_window // отзыв партии: также инцидент Pₛ PCF = clamp(0, 100, PCF) // просроченный сертификат → cert_band×0.5 (актуальность дублирует Dᵢ)
Пример

CDMO: действующий GMP (second-party, 75), валидация процесса полная (process_band 100), tamper-evident + чисто (integrity 100), 0 отзывов.

PCF = 0.45·75 + 0.30·100 + 0.25·100 = 33.75 + 30 + 25 = 88.75.

07.2 — SCIF: Supply Chain Integrity

Насколько раскрыта и верифицирована цепочка поставок. Доминирует у дистрибьюторов, telehealth, оптовиков, заводов. Blind-раскрытие (раздел 02 v2) засчитывается как валидное.

Входные сигналы

  • disclosed_ratio — доля партий с раскрытым и верифицированным (в т.ч. blind) источником.
  • depth_factor = min(1, verified_hops / target_hops) — глубина прослеженной цепочки (target ≈ 2 хопа до производителя).
  • broken_chain — партии с разорванной/противоречивой провенансой (также бьёт TRF/Pₛ).

Формула

SCIF_base = 100 · disclosed_ratio · (0.6 + 0.4·depth_factor) SCIF = clamp(0, 100, SCIF_base − 15·broken_chain_share·100)

Коэффициент 0.6/0.4 (база/глубина) и target_hops — дефолты под валидацию. Blind-источник входит в disclosed_ratio; пониженная доверенность отражается отдельно через Vᵢ (λ_blind=0.85), не в Fᵢ.

Пример

Оптовик: 90% партий с верифицированным источником, прослежен 1 хоп из 2 (depth_factor 0.5), broken_chain 0.

SCIF = 100·0.9·(0.6+0.4·0.5) = 90·0.8 = 72.

07.3 — FRF: Fulfillment Reliability

Надёжность исполнения обязательств: завершённость сделок, сроки, пропажи. Доминирует у таможенных брокеров, GB-организаторов, дропшипперов, аптек.

Входные сигналы

  • completed_ratio — доля успешно завершённых поставок/сделок (для customs — успешные растаможки, для GB — завершённые закупки).
  • on_time_ratio — доля в срок.
  • lost_funds — подтверждённые пропажи средств/товара (для GB; также Pₛ + потолок тира).

Банды по completed_ratio → base под валидацию

completed_ratiobase
≥ 0.98100
0.95–0.9885
0.90–0.9570
0.80–0.9045
< 0.8020
FRF = base · (0.7 + 0.3·on_time_ratio) − 25·lost_funds_incidents FRF = clamp(0, 100, FRF)
Пример

GB-организатор: completed 0.96 (base 85), on_time 0.9, 0 пропаж средств.

FRF = 85·(0.7+0.3·0.9) = 85·0.97 = 82.5. Одна подтверждённая пропажа средств → −25 + инцидент Pₛ.

07.4 — CCF: Cold Chain

Соблюдение холодовой цепи. Доминирует у курьеров последней мили и B2B-логистов; присутствует у RUO и аптек.

Входные сигналы

  • in_spec_ratio — доля поставок без температурных эксцессов (по мониторингу).
  • iot_coverage — доля поставок с IoT/датчик-логами (немониторимое = непроверяемо → дисконт).
  • breaches — нарушения цепочки с влиянием на продукт (также Pₛ при отрицании).

Формула

CCF_base = 100 · in_spec_ratio CCF = CCF_base · (0.5 + 0.5·iot_coverage) − 15·breaches_with_impact CCF = clamp(0, 100, CCF)

Самодекларация без IoT → iot_coverage низок → Fᵢ занижен; дополнительно Vᵢ=0.40 для self-declared (раздел 02 v2). Двойной демпфер намеренный: непроверяемая холодовая цепь почти не повышает Score.

Пример

Курьер последней мили: in_spec 0.95, iot_coverage 0.7, 0 нарушений.

CCF = 95·(0.5+0.5·0.7) = 95·0.85 = 80.75.

07.5 — CVF-B: Counterparty Verification (B2B)

B2B-аналог CVF: широта и подтверждённость верифицированных деловых связей. Доминирует у экспортных брокеров, таможенных брокеров, оптовиков, logистов, telehealth.

Входные сигналы

  • N_partners — число верифицированных B2B-контрагентов (с подтверждённой сделкой).
  • avg_partner_subscore — средний релевантный субскор партнёров (0–100).
  • partner_disputes — открытые споры с контрагентами (штраф).

Формула

confidence(N) = N / (N + k_b) // k_b ≈ 3 — под валидацию CVF_B = avg_partner_subscore · confidence(N_partners) − 12·partner_disputes CVF_B = clamp(0, 100, CVF_B)

Разведение с пропагацией: CVF-B — это фактор (широта/подтверждённость вашей B2B-сети). Перетекание доверия (раздел 05 v2) действует отдельно через Vᵢ. Чтобы не было двойного учёта: CVF-B считает «с кем вы верифицированно работаете», propagation — «насколько верить вашим данным от субскора контрагента».

Пример

Экспортный брокер: 8 верифицированных партнёров, средний субскор 78, k_b=3, 0 споров.

confidence = 8/11 = 0.73 → CVF-B = 78·0.73 = 56.9.

07.6 — RF: Responsiveness & Resolution

Скорость реакции и доля разрешённых инцидентов. Доминирует у таможенных брокеров, курьеров, инфлюенсеров; присутствует у лабораторий и логистов.

Входные сигналы

  • response_time — медианное время ответа на запрос/жалобу.
  • resolution_ratio — доля разрешённых инцидентов от всех.
  • reopened — повторно открытые («ложно закрытые») инциденты (штраф).

Банды под валидацию

response_time → response_bandresolution_ratio → resolution_band
≤ 24 ч → 100≥ 0.95 → 100
≤ 72 ч → 800.85–0.95 → 80
≤ 7 дн → 550.70–0.85 → 55
> 7 дн → 25< 0.70 → 30
RF = 0.4·response_band + 0.6·resolution_band − 10·reopened RF = clamp(0, 100, RF)
Пример

Таможенный брокер: медиана ответа 48 ч (band 80), resolution_ratio 0.9 (band 80), 0 reopened.

RF = 0.4·80 + 0.6·80 = 80.

Все 9 факторов — на одном уровне детализации

QEF, CVF, TRF (§04–06) + PCF, SCIF, FRF, CCF, CVF-B, RF (§07) теперь имеют входные сигналы, формулу, банды и пример. Любой Fᵢ для любой из 17 скорящихся ролей вычислим по матрице §08 и собирается в TS по формуле §09.

9 факторов реально задействованы. QEF и CVF — самые «сквозные» (встречаются у большинства ролей), что подтверждает их выбор в MVP-3 вместе с TRF (доминирует у посредников: брокеры, бренды, дропшипперы, инфлюенсеры). PCF и CCF — узкоспециализированы (производство и логистика). Это валидирует приоритет MVP-факторов.

09 — Сквозной пример TS

Полный расчёт для одной роли — эталон для разработки и QA. Берём RUO-вендора (веса: QEF 40, CVF 25, FRF 15, CCF 12, TRF 8). Fᵢ — из нормализации (§04–06), Vᵢ — из оракульной сети (раздел 02 v2), Dᵢ — из распада (раздел 03 v2).

Доминирующий фактор (по фактическому вкладу): QEF = 37.0 — явный лидер (следующий CVF = 9.9). Подпись: «61 — преимущественно качество». Тир: Silver (по порогам раздела 03 v2).

Что показывает пример

Идеальные 100 достижимы только при Fᵢ=100, Vᵢ=1, Dᵢ=1 по всем факторам — то есть лишь при blind-тестах и свежих данных везде. Реалистичный потолок ниже by design: Vᵢ и Dᵢ — это «налог на непроверяемость и несвежесть». Самодекларация (CCF, Vᵢ=0.40) почти не двигает Score — ровно как задумано.

Вариант со штрафом (для контраста)

Если у того же вендора подтверждён sample-swap (p=0.30, PK_SAMPLE_SWAP из Parameters Registry): TS = 60.97 × (1 − 0.30) ≈ 42.7 + жёсткий потолок тира Bronze (раздел 03 v2). Один доказанный обман обнуляет преимущество качества — обман не «растворяется» в хороших факторах.
