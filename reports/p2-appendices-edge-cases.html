00 — Контекст и принятые решения

P2: приложения и edge-кейсы

Закрывает оставшиеся P2 из аудита полноты: каденция пересчёта, ранжирование/тай-брейк, переход из Provisional, lifecycle участника (смена роли, merge/split, recall, оффбординг), lifecycle Steward/оракула, ER-модель, OpenAPI-контур, KYB doc-matrix, уведомления. После этого — сборка единого PRD.

01 — Каденция пересчёта

Когда Score пересчитывается

Гибрид: алерты не должны опаздывать, но полный пересчёт по decay для всех участников каждую секунду — лишняя нагрузка.

ТриггерРежим
Pₛ-флаг (мошенничество, sample-swap, алерт по контрагенту)Сразу (событийный пересчёт + анкоринг)
Новый независимый тест / артефактСразу
Отмена демпфера (продолжение работы с flagged-контрагентом)Сразу (см. v2 §«реакция на алерт»)
Decay (старение данных, Dᵢ)Батч раз в сутки
Пересчёт пропагации Vᵢ от соседейБатч раз в сутки (после стабилизации соседей)
Почему гибрид

Критичные негативные события (которые защищают покупателя) применяются мгновенно — это требование безопасности. Плавное старение можно считать батчем: оно предсказуемо и не требует реального времени. Так сохраняется и оперативность алертов, и разумная стоимость инфраструктуры.

02 — Ранжирование и тай-брейк

Порядок при равном Score в сегменте

Каталог сегментирован по ролям; внутри сегмента участники сортируются по Score. При равенстве нужен детерминированный и неэксплуатируемый порядок.

Каскад тай-брейка

  1. Покрытие Vᵢ (доказанность): выше тот, чей Score подтверждён большим числом независимых верификаций — равный балл, но более доказанный, ценнее.
  2. Свежесть данных: при равной доказанности — выше с более свежими подтверждениями.
  3. Детерминированный хеш-порядок: при полном равенстве — стабильный псевдослучайный порядок от hash(participant_id || epoch), исключающий ручной фаворитизм.
Анти-гейминг

Тай-брейк по покрытию Vᵢ нельзя накрутить дешёвыми self-сигналами: учитываются только независимые подтверждения (blind/random тесты, verified-покупатели). Хеш-порядок убирает мотив «дожать» позицию манипуляцией на последнем шаге.

03 — Provisional → полный Score

Критерий выхода — доказательства, не календарь

New/Provisional участник получает полноценный Trust Score, когда накопил достаточно проверяемых данных, а не просто «прожил» срок.

УсловиеПорог (provisional v0.9, под SME)
KYB/KYC статусL2 Verified Business (или fallback L1 + потолок — см. P1 §02)
Независимые подтверждения по ключевым факторам≥ N подтверждений (напр. ≥2 независимых источника по доминирующему фактору роли)
Отсутствие открытых критичных Pₛ-флаговНет активных CRITICAL-флагов
Почему по доказательствам

Календарный срок поощряет «отсидеться» без реальных данных и эксплуатируется фабриками новых аккаунтов. Привязка к независимым подтверждениям делает переход функцией доказанного поведения. Конкретное N — параметр под валидацию SME (чек-лист в P0-closure).

04 — Lifecycle участника

Смена роли, merge/split, recall, оффбординг

Смена роли

  • Score не переносится напрямую между ролями: веса Wᵢ и доминирующие факторы различаются. История сохраняется и помечается прежней ролью; новая роль стартует с Provisional по своим факторам.
  • Факторы, общие для обеих ролей (напр. логистика CCF), могут переноситься как подтверждённые входы с сохранением Dᵢ-старения.

Merge / Split (объединение / разделение участников)

  • Merge: при доказанной тождественности (один бизнес под двумя профилями) — объединение с сохранением худшего из активных Pₛ-флагов (нельзя «смыть» негатив через слияние).
  • Split: при разделении бизнеса — новые сущности стартуют как Provisional; подтверждённые флаги наследуются стороной-правопреемником, чтобы исключить уход от ответственности через дробление.

Recall (отзыв партии/COA)

  • Отзыв партии или COA — событийный триггер: затронутый lot помечается recalled, связанные подтверждения теряют вес, пересчёт сразу.
  • Recall логируется публично; повторяющиеся recall повышают Pₛ.

Оффбординг (уход участника)

ЧтоСудьба
Профиль в каталогеПомечается «ушёл» (tombstone), скрыт из активной выдачи
Исторический Score + подтверждённые Pₛ-флагиОстаются публичными (защита третьих лиц), привязаны к algo_version
Персональные данныеМинимизируются/удаляются по data-protection режиму (P1 §04)
Ончейн-хешиСохраняются для целостности (неизменяемы)
Принцип

Уход не должен быть способом стереть подтверждённый негатив. Поэтому tombstone сохраняет публично значимую историю (Score, фрод-флаги), но снимает лишние ПДн. Это баланс права на забвение и защиты будущих покупателей.

05 — Lifecycle Steward / оракула

Назначение, срок, ротация, снятие

Steward'ы (участники апелляционных панелей) и Verified-оракулы (лаборатории) — доверенные роли; им нужен прозрачный жизненный цикл во избежание захвата.

ЭтапМеханика
НазначениеSteward — из пула участников по критериям (см. v2 §07 арбитраж); оракул — через аккредитацию Root CA (Волна 2)
СрокФиксированный с ротацией — исключает «вечную» концентрацию влияния
Снятие по триггерамКонфликт интересов, нарушение протокола, неактивность, подтверждённая необъективность
Отзыв ключей оракулаCRL / ротация ключей (Волна 2 PKI) при компрометации или деаккредитации
ПрозрачностьВсе назначения/снятия — в публичном логе управления
Почему срок + триггеры

Бессрочные доверенные роли тяготеют к захвату и сговору. Срок с ротацией обновляет состав, триггеры снимают недобросовестных досрочно, а публичный лог делает процесс проверяемым. Это согласуется с VRF-отбором панелей и cross-segment принципом из v2.

06 — ER-модель данных

Ключевые сущности и связи

Логическая модель (не финальная схема БД) — основа для PRD-приложения и OpenAPI.

Participant (id, role, status[Provisional|Active|Tombstone], kyb_level, algo_version) └─1:N─ ScoreEvent (id, participant_id, ts, score, dominant_factor, algo_version, anchor_hash) └─1:N─ FactorInput (id, participant_id, factor_code, value, Dᵢ_class, Vᵢ, source_id, ts) └─1:N─ PenaltyFlag (id, participant_id, type, severity[INFO|WARNING|CRITICAL], pₖ, opened_at, status) Lot (id, vendor_id, status[active|recalled], lot_qr_key) └─1:N─ CustodyRecord (lot_ref, sampler_id, steps[], lab_id, sig_lab, anchor_hash) └─1:N─ COA (id, lot_ref, lab_id, signed_payload, anchor_hash) Source/Oracle (id, type[buyer|GB|auditor|lab], independence_flag, accreditation) Counterparty-Link (from_id, to_id, lot_share, blind_flag, λ) // для пропагации Vᵢ Appeal (id, subject_participant, panel[], chair_oracle, decision, cost_bearer) Steward (participant_id, term_start, term_end, status) GovernanceLog (id, event, actor, ts, anchor_hash)

Связи поддерживают всё: пропагацию (Counterparty-Link), доказанность (FactorInput.Vᵢ + Source.independence), оффбординг (Participant.status), арбитраж (Appeal) и аудит (GovernanceLog/anchor_hash).

10 — Что дальше

P2 закрыты — остаётся сборка PRD

  1. Каденция (§01) — гибрид: события сразу, decay батчем.
  2. Ранжирование (§02) — каскад: покрытие Vᵢ → свежесть → хеш-порядок.
  3. Provisional → полный (§03) — по доказательствам (KYB + N подтверждений).
  4. Lifecycle участника (§04) — смена роли, merge/split, recall, tombstone-оффбординг.
  5. Lifecycle Steward/оракула (§05) — срок + ротация + триггеры снятия.
  6. ER-модель, OpenAPI-контур, KYB doc-matrix, уведомления (§06–09) — приложения для PRD.
Готовность к PRD

Закрыты все P0, P1 и P2. Следующий и финальный шаг — сборка единого PRD-документа: свести Волны 0–4, закрытия P0/P1/P2 и v2-реестр в один структурированный документ с разделами (видение, роли, алгоритм, идентичность, ончейн, продукт, UX, безопасность, операции, экономика, право, приложения). Открытые внешние зависимости остаются как явные «to be confirmed»: числа экономики, подпись SME, юрист (ToS/defamation), DPO (DPIA/состав хешей).

Статус на дату ревизии: описанный «финальный шаг» (сборка единого PRD) впоследствии выполнен — это исторический phase-closure отчёт. Актуальный сводный статус — в PRD.

P2-решения — инженерные/продуктовые каркасы; параметры (N подтверждений, срок Steward, пороги) — под валидацию SME из чек-листа P0-closure.
