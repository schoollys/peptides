PRD Foundations — сшивающий слой перед сборкой PRD

Глоссарий, NFR и реестр допущений

Три «сшивающих» приложения, обеспечивающих связность единого PRD: единый словарь терминов, консолидированные нефункциональные требования и честный реестр открытых вопросов (to-be-confirmed). Рабочее название продукта — PeptideTrust (placeholder под возможный ребрендинг).

01 — Канонический глоссарий

Единственный источник истины по терминам; все остальные документы должны соответствовать этим определениям.

Алгоритм и оценка

ТерминОпределение
Trust Score (TS)Итоговый публичный балл доверия: TS = Σ(Fᵢ·Wᵢ·Dᵢ·Vᵢ)·(1−Pₛ)
FᵢFactor score (0–100) — внутреннее качество/результат по фактору, отделён от надёжности источника
WᵢВес фактора для роли, ΣWᵢ=1.0
DᵢМножитель свежести (decay); 4 класса с экспоненциальным затуханием (полупериоды A:365 B:120 C:45 D:90 дней)
VᵢМножитель верификации: 0.40 + (Sub/100)·0.55; λ_blind=0.85 для blind-связей; пропагация одношаговая
PₛМножитель штрафов: 1 − ∏(1−pₖ) с жёсткими потолками тиров
Доминирующий факторФактор с наибольшим вкладом (Fᵢ·Wᵢ·Dᵢ·Vᵢ); показывается рядом со Score
Факторы (9)QEF, PCF, SCIF, TRF, FRF, CCF, CVF, CVF-B, RF (см. словарь факторов в v2)
algo_versionМонотонный идентификатор набора параметров (веса/банды/штрафы/Dᵢ/Vᵢ); анкорится в L2; forward-only

Верификация и данные

ТерминОпределение
Оракул (Verified Oracle)Аккредитованный источник доказательств (независимая лаборатория) с PKI-подписью
Естественный оракулУчастник, по роли отбирающий образец независимо от вендора (GB-организатор, покупатель)
Blind / shielded disclosureДоказать факт без раскрытия личности апстрима; по умолчанию для посредников; λ_blind=0.85
Chain-of-custodyПодписанная на каждом шаге цепочка владения образцом; неполнота → тест не = Vᵢ 1.0
Sample-swapПодмена образца вендором; бьёт по Pₛ вендора, не по лаборатории
Trust-пропагацияОдношаговый перенос профильного субскора контрагента в Vᵢ (не всего TS)
Lot-QRКриптоключ партии для привязки COA/артефактов к конкретному лоту
tombstoneРежим удаления: ПДн снимаются, публично значимая история (Score, фрод-флаги) и ончейн-хеши сохраняются

02 — Нефункциональные требования

NFR единым блоком

Консолидация нефункциональных требований из Волн 3–4 в единый блок для PRD. Числовые таргеты — стартовые (provisional), уточняются по нагрузке.

КатегорияТребование (старт)
ПроизводительностьЧтение профиля/каталога p95 < 300 мс; бейдж (кэшируемый SVG) p95 < 150 мс
ДоступностьПубличное read-API и бейдж — целевой uptime 99.9%; деградация только write-путей при инцидентах
МасштабируемостьStateless read-слой за CDN; батч-пересчёт decay масштабируется по числу профилей; событийный пересчёт — очередь
БезопасностьEU/CH хостинг; ключи в KMS/HSM; PKI для оракулов/write; anti-DDoS; payload-подпись на write (Волна 4)
ПриватностьPrivacy-by-design; анонимные ончейн-хеши (соль, без идентификаторов); минимизация ПДн; DPIA до запуска
ЦелостностьОнчейн-якоря (L2 Base/Arbitrum) для Score-событий, COA, custody; публичная верификация по anchor_hash
НаблюдаемостьМетрики/логи/трейсинг; публичный лог инцидентов и управления; алертинг по аномалиям
Доступность интерфейса (a11y)Каталог/профиль/бейдж — WCAG 2.1 AA как цель; бейдж с alt/контрастом
СовместимостьБейдж как server-rendered image — работает без JS на любом сайте; современные браузеры для веб-UI
ЛокализацияMVP — английский (глобальный RUO-рынок); i18n-готовность архитектуры, локали отложены до V2
SLA восстановленияСтартовые цели реакции/восстановления по severity (Волна 4); RPO/RTO уточняются
Принцип

Read-путь (каталог, профиль, бейдж, верификация) — критичный и должен быть быстрым/высокодоступным даже при деградации write. Это прямо следует из pull-стратегии: ценность для спроса не должна падать из-за проблем на стороне подачи данных.

Зачем это в PRD

Зрелый PRD не прячет неизвестное, а явно перечисляет допущения с владельцами. Это превращает «дыры» в управляемый бэклог решений и защищает от ложного ощущения завершённости. P0-зависимости (SME, экономика, юрист, DPO) — блокеры продакшена, не MVP-демо.

04 — Что дальше

Сшивающий слой готов → сборка PRD

Добавлены три приложения: глоссарий (§01), NFR (§02), реестр допущений (§03). Название зафиксировано как рабочий placeholder. Формат PRD — основной документ + ссылки на детальные приложения (Волны, закрытия), без дублирования.

Финальный шаг

Теперь можно собирать единый PRD: краткий самодостаточный core-документ (видение → требования → архитектура → продукт → операции → метрики → допущения) со ссылками на это приложение, Vision, Волны 0–4, закрытия P0/P1/P2 и v2-реестр как источники глубины.
