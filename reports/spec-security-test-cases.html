Security Test Cases

Конкретные тест-кейсы (Given/When/Then), выведенные из threat model (STRIDE + 8 нишевых abuse-сценариев) и PKI/AuthZ/privacy. Питают раздел Security Test Strategy; каждая угроза имеет ≥1 кейс.

Spoofing
SEC-S1Поддельная подпись на writeSpoofing

Given валидный payload и подделанный/чужой X-Signature

When POST /submissions

Then 401 BAD_SIGNATURE; запись не создана; событие в аудит-лог

SEC-S2COA от неаккредитованного «оракула»Spoofing

Given COA, подписанный ключом без записи в Oracle-PKI

When подача kind=coa

Then отклонение; вклад не засчитан; сертификат не в доверенной цепочке

Tampering
SEC-T1Подмена Score задним числомTampering

Given заанкоренное событие Score с anchor_hash

When попытка изменить score-событие в БД и пересобрать

Then /verify/{hash} не сходится с L2; расхождение детектируется

SEC-T2Подмена тела бейджаTampering

Given подписанный SVG-бейдж

When изменение числа Score в SVG на стороне сайта

Then подпись невалидна; клик ведёт на профиль с ончейн-проверкой (расхождение видно)

Repudiation
SEC-R1Отрицание факта подачиRepudiation

Given участник подал подписанные данные с таймстампом

When участник отрицает подачу

Then аудит-лог + подпись + ончейн-якорь времени доказывают факт и момент

Information disclosure
SEC-I1Утечка shielded-идентичности через APIInfo disclosure

Given участник в режиме Verified-but-shielded

When запрос профиля/каталога/бейджа любым клиентом

Then real_identity и ПДн отсутствуют в ответе, логах и метаданных SVG

SEC-I2ПДн в логах/ошибкахInfo disclosure

Given ошибка валидации с пользовательскими данными

When формируется ответ ошибки и лог

Then ПДн маскированы; в ответе только code/message/request_id

Denial of service
SEC-D1Флуд на read/бейджDoS

Given всплеск запросов с источника сверх лимита

When массовые GET /badge и /participants

Then CDN/кэш поглощает; rate-limit → 429 + Retry-After; ядро не деградирует

SEC-D2Спам апелляциями/репортамиDoS

Given участник с низкой reliability-of-initiator

When массовая подача репортов/апелляций

Then троттлинг по reliability; нет влияния на Score целей без подтверждения

Elevation of privilege
SEC-E1Несанкционированное изменение весовEoP

Given аккаунт без прав комитета

When попытка изменить role_factor_weights/algo_version

Then отказ; изменение возможно только через supermajority комитета + запись в changelog

Abuse · Sybil
SEC-A1Массовые фейковые аккаунты/отзывыSybil

Given попытка создать множество L0 с одного устройства/номера

When регистрация + безартефактные отзывы

Then PoP (phone+device) блокирует дубли; безартефактный отзыв вес 0.45; нет KYB → trust ceiling

Abuse · Сговор
SEC-A2Кольцо взаимной накрутки VᵢCollusion

Given группа аффилированных участников, ссылающихся друг на друга

When взаимные подтверждения/пропагация по циклу

Then CYCLE_DETECTED — пропагация запрещена; граф-детектор помечает кластер; концентрация top_share>0.6 штрафуется

Abuse · Sample-swap
SEC-A3Чистый образец на тест, разбавленный в продажуSample-swap

Given вендор сам предоставляет образец на тест (self-sampling)

When подаётся COA с высоким Fᵢ

Then Vᵢ<1.0 (не независимый сэмплинг); при выявлении подмены — Pₛ-удар вендору, не оракулу

Abuse · Компрометация ключа оракула
SEC-A4Подписи после компрометацииKey compromise

Given сертификат оракула отозван (CRL) с датой компрометации

When проверяются подписи до и после этой даты

Then подписи ДО — валидны; ПОСЛЕ — отклонены; вклад снят, пересчёт (см. RB-07)

Abuse · Реюз артефактов
SEC-A5Повторная подача того же медиаArtifact reuse

Given медиа-артефакт, уже использованный (media_hash/pHash известен)

When повторная подача (тот же или дубликат-изображение)

Then ARTIFACT_REUSE — отклонение; nonce-челлендж/EXIF-проверка; нет повторного вклада

Abuse · Атака репутацией
SEC-A6Фейк-репорты, чтобы «утопить» конкурентаReputation attack

Given поток неподтверждённых негативных репортов на участника

When репорты обрабатываются

Then буфер демпфера НЕ аннулируется без подтверждения; reliability-of-initiator снижает вес; спорное → Steward-панель

Abuse · Стирание истории
SEC-A7Сброс негатива через удаление и пересозданиеHistory erase

Given участник с подтверждёнными Pₛ-флагами запрашивает удаление

When оффбординг и попытка вернуться «с нуля»

Then tombstone: история/флаги публичны (ПДн сняты); возврат → повторный KYB + старт с Provisional

Abuse · Подделка бейджа
SEC-A8Чужой/устаревший бейдж на сайтеBadge forgery

Given сайт встраивает скопированный/устаревший бейдж

When пользователь видит бейдж и кликает

Then серверный рендер с TTL отдаёт актуальное значение; клик → профиль с ончейн-проверкой; подделка не проходит подпись

Кросс · AuthZ / доступ
SEC-X1Доступ к чужому write-ресурсуAuthZ

Given валидный API-key участника A

When попытка подать/изменить данные от имени участника B

Then 403 FORBIDDEN по матрице доступа (ERD & Security); запись в аудит

SEC-X2Replay вебхука/запросаAuthZ/Integrity

Given перехваченный валидный запрос с таймстампом

When повтор спустя время

Then устаревший X-PT-Timestamp отклонён; идемпотентность по delivery_id/Idempotency-Key

Связка

Каждый кейс ссылается на контрмеру в threat model и/или бизнес-правило (BR-*). Runbooks (ops-runbooks.html) описывают реакцию, когда контрмера всё же пробита.
