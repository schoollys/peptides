User Stories и Acceptance Criteria

Истории сгруппированы по эпикам и фазам (MVP / V1 / V2). Каждая история — формат «Как <роль>, я хочу <цель>, чтобы <ценность>» с критериями приёмки (AC) в стиле Given/When/Then. Покрытие — все фазы; MVP детально, V1/V2 — ключевые истории каждого эпика. Персоны из Vision §03.

Числовые пороги в AC (N подтверждений, тайминги, лимиты) — провизорные, ссылаются на чек-лист SME (P0-closure) и каталог бизнес-правил.

E1 — Каталог и просмотр профиля
US-1.1
Проверка поставщика перед сделкой
Как покупатель, я хочу за минуту найти поставщика и увидеть его доверие, чтобы не рисковать здоровьем и деньгами.
Acceptance Criteria
  • Given я открыл каталог без регистрации, When ищу поставщика по имени/домену, Then вижу его профиль со Score и доминирующим фактором.
  • Given поставщик не зарегистрирован, When я его ищу, Then вижу либо Unclaimed-профиль (нейтральные факты), либо явное «нет данных».
  • Загрузка профиля p95 < 300 мс; каталог доступен анонимно.
US-1.2
Сравнение в сегменте
Как покупатель, я хочу сравнить нескольких поставщиков одной роли, чтобы выбрать наиболее проверенного.
Acceptance Criteria
  • Given сегмент «вендоры RUO», When применяю фильтры (тир, доминирующий фактор), Then список сортируется по Score с тай-брейком по покрытию Vᵢ.
  • Given равный Score, When вижу порядок, Then выше тот, у кого больше независимых подтверждений (см. бизнес-правила BR-RANK).
  • Verified-Legal каналы отделены фильтром.
E2 — Score и доминирующий фактор
US-2.1
Понятный Score с контекстом
Как покупатель, я хочу видеть не абстрактное число, а по какому измерению участник силён, чтобы правильно интерпретировать балл.
Acceptance Criteria
  • Given профиль со Score 82, When смотрю карточку, Then вижу «82 — преимущественно качество (QEF)» с доминирующим фактором.
  • Given сбалансированный профиль, When нет явного лидера, Then подпись «сбалансированный профиль» (см. BR-DOMFACTOR).
  • Given роль без полного TS, When смотрю, Then вижу репутационную заметку без числового Score.
  • Доминирующий фактор = max вклад (Fᵢ·Wᵢ·Dᵢ·Vᵢ); глоссарий из 9 факторов доступен.
E3 — Бейдж и публичное API
US-3.1
Встраиваемый бейдж на сайт
Как вендор, я хочу встроить бейдж со своим Score, чтобы повысить конверсию доверием.
Acceptance Criteria
  • Given у меня есть профиль, When копирую сниппет бейджа, Then на моём сайте отображается актуальный подписанный SVG/PNG без JS.
  • Given бейдж встроен, When кто-то кликает, Then ведёт на профиль и фиксируется клик-аналитика.
  • Given попытка подделать значение в бейдже, When рендер запрашивается, Then подпись эндпоинта не проходит верификацию → бейдж невалиден.
  • Рендер бейджа p95 < 150 мс (кэшируемый); tier-gating визуала по тиру.
US-3.2
Публичное read-API
Как платформа/разработчик, я хочу читать Score через API, чтобы интегрировать проверку у себя.
Acceptance Criteria
  • Given публичный эндпоинт, When запрашиваю GET /v1/participants/{id}, Then получаю Score, доминирующий фактор, статус, флаги.
  • Given anchor_hash, When вызываю GET /v1/verify/{hash}, Then подтверждается целостность ончейн-записи.
  • Версионирование /v1; rate-limit; ответы кэшируемы.
E4 — Онбординг и идентичность
US-4.1
Claim профиля
Как вендор, я хочу заявить и верифицировать свой профиль, чтобы управлять данными и получить полноценный Score.
Acceptance Criteria
  • Given существует Unclaimed-профиль, When я подаю claim + прохожу KYB, Then профиль переходит в управляемый и получает статус New/Provisional.
  • Given моя юрисдикция не поддерживает KYB, When прохожу fallback L1 (KYC), Then получаю профиль с потолком доверия и упором на blind-тесты (BR-KYB-FALLBACK).
  • Given claim на чужой профиль, When не прохожу верификацию владения, Then claim отклоняется.
US-4.2
Fast-track для новичка
Как новый вендор, я хочу быстро стартовать, чтобы не ждать месяцами накопления репутации.
Acceptance Criteria
  • Given прошёл KYB + 1 blind-тест, When завершаю онбординг, Then получаю статус Provisional с видимостью в каталоге.
  • Given накопил KYB + N независимых подтверждений, When порог достигнут, Then перехожу в полный Trust Score (BR-PROVISIONAL, по доказательствам, не календарю).
US-4.3
Анти-Sybil на L0
Как система, я хочу ограничить массовую накрутку аккаунтов, чтобы L0-сигналы не подделывались дёшево.
Acceptance Criteria
  • Given регистрация L0, When создаётся аккаунт, Then требуется уникальный телефон + device/CAPTCHA-проверка.
  • Given аномальный паттерн мультиаккаунтов, When детектор срабатывает, Then аккаунты помечаются для ревью.
E5 — Тесты, оракулы, ончейн
US-5.1
Подписанный COA от оракула
Как лаборатория-оракул, я хочу выдавать неподделываемые заключения, чтобы защитить репутацию и получать заказы.
Acceptance Criteria
  • Given я аккредитованный оракул, When публикую COA, Then он подписан моим PKI-ключом и привязан к партии.
  • Given независимый отбор образца (custody полна), When COA принят, Then фактор получает Vᵢ=1.0.
  • Given образец предоставлен вендором, When COA принят, Then Vᵢ ≤ 0.70 (vendor-grade), не 1.0.
US-5.2
Ончейн-якорь целостности
Как покупатель/аудитор, я хочу проверить, что данные не подменены задним числом, чтобы доверять истории.
Acceptance Criteria
  • Given событие Score/COA/custody, When оно зафиксировано, Then хеш анкорится в L2 с algo_version.
  • Given попытка изменить историю, When сверяю с ончейн-хешем, Then расхождение обнаруживается.
  • В ончейн идут только анонимные хеши (соль, без идентификаторов).

E6 — Trust-пропагация и алерты
US-6.1
Влияние контрагентов на доверие
Как покупатель, я хочу, чтобы доверие учитывало надёжность контрагентов участника, чтобы видеть системный риск.
Acceptance Criteria
  • Given участник работает с контрагентами, When считается Vᵢ, Then применяется одношаговая пропагация профильного субскора (не всего TS) с λ_blind для blind-связей.
  • Given концентрация источников (top_share > 0.60), When агрегируется V_prop, Then применяется штраф за концентрацию и потолок 0.85 (BR-PROP).
  • Given циклическая связь, When считается пропагация, Then цикл исключается.
US-6.2
Алерт и демпфер
Как вендор, я хочу получать алерт о падении контрагента и буфер на реакцию, чтобы успеть среагировать.
Acceptance Criteria
  • Given резкое падение субскора контрагента, When событие зафиксировано, Then я получаю алерт, а влияние сглаживается 30-дневным средним.
  • Given я получил алерт, When продолжаю новое сорсинг у flagged-контрагента, Then 30-дневный буфер отменяется и Score пересчитывается вниз сразу (BR-DAMPER).
E7 — Chain Pull
US-7.1
Втягивание контрагентов
Как участник, я хочу пригласить контрагентов в реестр, чтобы повысить проверенность своих связей и получить кредит.
Acceptance Criteria
  • Given я приглашаю контрагента, When он регистрируется и верифицируется, Then мне начисляется Chain Pull-кредит, а связь становится верифицируемой.
  • Given контрагент не в реестре, When покупатель смотрит мой профиль, Then отсутствие связей-подтверждений видно как сниженное покрытие Vᵢ.
E8 — Апелляции и Steward-панели
US-8.1
Оспаривание спорного решения
Как участник, я хочу оспорить несправедливый флаг/отзыв, чтобы защититься от ошибок и атак.
Acceptance Criteria
  • Given я не согласен с флагом, When подаю апелляцию, Then формируется cross-segment панель Steward'ов через VRF-жеребьёвку с conflict-screening.
  • Given панель собрана, When есть конфликт интересов у кандидата, Then он отсеивается графовым скринингом; председатель — Verified Oracle.
  • Given решение вынесено, When оно опубликовано, Then оно обязательно в рамках реестра; издержки по loser-pays с защитой добросовестности; правовой путь сохранён.
E9 — B2B-продукты
US-9.1
Due-diligence отчёт
Как клиника/дистрибьютор, я хочу глубокий отчёт о поставщике, чтобы снизить риск закупки.
Acceptance Criteria
  • Given я B2B-клиент, When заказываю due-diligence по поставщику, Then получаю историю факторов, цепочку, флаги, динамику.
  • Given оцениваемый поставщик, When он хочет заплатить за влияние на отчёт, Then это невозможно (BR-NEUTRALITY).
US-9.2
Мониторинг-API
Как B2B-клиент, я хочу подписаться на изменения Score контрагентов, чтобы реагировать вовремя.
Acceptance Criteria
  • Given подписка на контрагента, When его Score/флаг меняется, Then приходит webhook с событием.
  • Write/protected эндпоинты — API-key + payload-подпись.
E10 — Lot-QR и chain-of-custody
US-10.1
Привязка теста к партии
Как покупатель, я хочу убедиться, что COA относится именно к этой партии, чтобы не доверять «общему» сертификату.
Acceptance Criteria
  • Given партия с lot-QR, When сканирую/проверяю, Then вижу привязанные к лоту COA и custody-цепочку.
  • Given разрыв подписи на шаге custody, When тест оценивается, Then он не получает Vᵢ=1.0 (BR-CUSTODY).
  • Given попытка повторного использования артефакта, When он подаётся снова, Then анти-reuse детект отклоняет.

E11 — ZK и приватность
US-11.1
Доказательство без раскрытия
Как посредник, я хочу доказать надёжность апстрима без раскрытия его личности, чтобы сохранить коммерческую тайну.
Acceptance Criteria
  • Given blind-связь, When формируется доказательство, Then ZK-доказательство подтверждает факт без раскрытия идентичности (замена shielded assertion MVP).
  • Given downstream-сторона, When управляет видимостью, Then контролирует раскрытие своих связей.
E12 — Аналитика, VaaS, охват
US-12.1
Рыночная аналитика
Как исследователь/инвестор, я хочу агрегированные тренды ниши, чтобы принимать решения по рынку.
Acceptance Criteria
  • Given аналитика-SKU, When запрашиваю тренды, Then получаю только агрегированные данные (без продажи влияния на конкретный Score).
US-12.2
Verification-as-a-service
Как сторонняя платформа, я хочу встроить проверку/бейдж к себе, чтобы дать пользователям доверие.
Acceptance Criteria
  • Given VaaS-интеграция, When подключаю SDK/API, Then могу показывать верифицированный Score у себя с сохранением нейтралитета и подписи.
