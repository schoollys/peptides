TDD: Sequence Diagrams & State Machines

Детализация сложных модулей: последовательности взаимодействий и конечные автоматы жизненных циклов. Дополняет SAD (потоки) и Business Rules (переходы). Диаграммы текстовые (ASCII) — переносимы в любой инструмент.

Ключевое

Write возвращает 202 сразу; тяжёлый пересчёт и anchoring — асинхронно (ADR-010). Идемпотентность по Idempotency-Key защищает от дублей при ретрае.

06 — State Machine: статус участника
unclaimed ──claim+KYB──► provisional ──(KYB + N незав. подтв., нет CRITICAL)──► active │ │ │ │ │ CRITICAL flag│ │ ▼ ▼ │ (остаётся provisional suspended │ при недоборе) │ │ разрешение/апелляция └────────────────────────── offboard ─────────────────────────► tombstone ◄──┘ // tombstone: ПДн сняты, история/флаги публичны (BR-OFFBOARD)

Переход provisional→active — по доказательствам, не по календарю (BR-PROVISIONAL).

active ──recall (отзыв партии/COA)──► recalled │ │ │ COA/тесты привязаны (lot-QR) │ связанные подтверждения теряют вес │ │ пересчёт сразу + Pₛ при повторах ▼ ▼ учитывается в Score исключён из положительного вклада (BR-RECALL)

09 — State Machine: anchoring в L2
pending ──submit tx──► (ожидание подтверждения) │ │ │ ошибка/таймаут │ confirmed ▼ ▼ failed ──retry (backoff)──► anchored // failed не блокирует read; ретраи с экспоненциальным backoff
Принцип TDD

Каждый сложный модуль имеет явный автомат состояний и последовательность. Это убирает неоднозначность реализации и связывает код с бизнес-правилами (BR-*) и acceptance criteria (US-*).
