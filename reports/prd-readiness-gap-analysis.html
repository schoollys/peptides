00 — Резюме и метод

Анализ готовности к PRD: что недораскрыто в документе v2

Документ v2-peptide-trust-registry.html — это сильный концептуальный и стратегический дизайн: он отлично отвечает на вопросы «зачем», «почему так» и «какая философия». Но PRD требует ответов на «что именно строим», «по каким точным правилам», «какие экраны, API, данные и параметры». Ниже — систематический разбор по 13 доменам PRD: что в документе уже есть, чего не хватает, и конкретная рекомендация по каждому пробелу.

~40
открытых вопросов к уточнению

Короткий вывод

Документ закрывает «видение и механику доверия» примерно на 85%. Для исчерпывающего PRD критически недостаёт четырёх вещей: (1) как сырые данные превращаются в баллы факторов Fᵢ (нормализация) и полная матрица весов роль×фактор; (2) модель данных, идентичности (KYB/KYC) и верификации артефактов покупки; (3) управление, версионирование параметров алгоритма и модель финансирования (кто платит за хостинг, газ, арбитров); (4) юридический/комплаенс-слой (ответственность за рейтинг в серой зоне, GDPR vs неизменяемость). Остальное — это полнота спецификаций (UX-экраны, API, метрики), которая достраивается уже на основе принятых принципов.

Легенда приоритетов: P0 блокер для PRD · P1 важно, влияет на архитектуру · P2 можно зафиксировать допущением и уточнить позже.

Статус на дату ревизии — пробелы закрыты

Это раунд-1 gap-снимок документа v2 (самый ранний аудит). Перечисленные ниже пробелы впоследствии закрыты: стратегия — Волна 0; ядро алгоритма (нормализация Fᵢ, матрица весов 22×9, субскор, Pₛ, worked example) — Волна 1 и Algorithm Spec; идентичность/ончейн/PKI/ZK — Волна 2; продукт/API/каталог — Волна 3; UX/безопасность/операции — Волна 4; 5 P0/12 P1/13 P2 — закрытия P0/P1/P2; сводно — PRD. Реально открыты лишь внешние зависимости (SME, юрист, DPO, числа экономики) — реестр PRD §14.

01 — Продукт и скоуп

Границы продукта, релизы и критерии готовности

P0Нет измеримого problem statement и определения успеха

Есть:Позиционирование «реестр, не маркетплейс», аналоги (Carfax, D&B), описание болей рынка.

Нет:Формулировки проблемы в измеримых терминах (масштаб рынка, % контрафакта, объём аудитории) и определения, что считается успехом продукта.

Рекомендация:Добавить раздел «Проблема и цель» с количественными ориентирами и North-Star-метрикой (см. §11).
P1Скоуп MVP не отделён от «вне скоупа»

Есть:Дорожная карта по фазам (MVP → V3) с фичами на каждой.

Нет:Явного списка «в скоупе / вне скоупа» для первого релиза, целевой географии и юрисдикции запуска, целевого сегмента (какие из 22 ролей онбордим первыми и почему).

Рекомендация:Зафиксировать MVP-scope таблицей in/out, выбрать 1 географию запуска и 3–4 «якорные» роли.
P1У фаз нет критериев приёмки (acceptance criteria)

Нет:Условий «фаза считается завершённой, если…» — каждая фаза описана как набор фич без порогов и Definition of Done.

Рекомендация:К каждой фазе добавить 3–5 проверяемых критериев (напр. «≥15 онбордингов, ≥3 оракула, время разрешения апелляции уровня 1 < 1 мин»).
02 — Пользователи и сценарии

Персоны, JTBD, user stories и приоритизация

P0Роли описаны как «боли/выгоды», но нет user stories и journey

Есть:22 роли с болями, выгодами, данными для подачи, весами факторов — очень богатая база персон.

Нет:Формальных user stories («как X, я хочу Y, чтобы Z»), пользовательских путей (онбординг → подача данных → получение Score → встраивание бейджа → апелляция) и приоритизации персон для MVP.

Рекомендация:Превратить каждую роль в набор user stories с acceptance criteria; построить journey map для 3–4 якорных ролей MVP.
P1Не описан онбординг и первичная регистрация

Нет:Сценария регистрации участника: какие шаги, какие данные обязательны, как проходит KYB/KYC, что видит пустой профиль, как присваивается роль (само-выбор? верификация?).

Рекомендация:Описать flow онбординга отдельно для бизнес-ролей (KYB) и для community-ролей (псевдонимная идентичность).
03 — Функциональные требования

Каталог, профиль, подача данных, бейдж, API

P0Каталог как ядро продукта не специфицирован

Есть:«Открытый каталог с фильтрами по категории/региону/Score, сравнение, сортировка по Score».

Нет:Полного списка фильтров и фасетов, правил ранжирования (что при равных Score? как учитывается доминирующий фактор, New/Provisional, Verified-Legal сегмент?), пагинации, дедупликации, поведения для ролей без TS, поиска по продукту/пептиду.

Рекомендация:Специфицировать каталог как отдельную фичу: модель сортировки, фасеты, дефолтные представления, edge-состояния.
P1Public Verify API описан одним эндпоинтом

Есть:GET /trust/{id} → JSON audit trail; бейдж и виджеты.

Нет:Полной спецификации: список эндпоинтов (каталог, профиль, история, сравнение, оракул-публикация, submission), схемы ответов, аутентификация для записи, rate limits, версионирование, форматы бейджа (размеры, темы, кеширование), защита live-бейджа от подделки/DDoS.

Рекомендация:Подготовить OpenAPI-черновик: публичные read-эндпоинты + защищённые write-эндпоинты (оракулы, участники).
P1Формы подачи данных и личный кабинет не описаны как фичи

Есть:Дашборд упоминается (полнота профиля, decay-уведомления, аналитика обращений, конкурентный контекст, clickthrough).

Нет:Какие именно формы подачи (COA, тест, IoT, раскрытие партии, отзыв), их поля, валидация, процесс хеширования при загрузке, состояния «на проверке/подтверждено/отклонено».

Рекомендация:Описать ЛК как набор модулей и формы submission с их полями и состояниями.
P2Уведомления (алерты) без спецификации каналов

Нет:Каналов (email/push/in-app/Telegram?), частоты, шаблонов, согласий — при том что вся механика давления (decay, алерты о падении контрагента, reaction window) держится на уведомлениях.

Рекомендация:Описать систему уведомлений: типы событий, каналы, частота, opt-in.
04 — Алгоритм: точные параметры

Самый важный технический пробел

Формула TS = Σ(Fᵢ·Wᵢ·Dᵢ·Vᵢ)·(1−Pₛ) определена концептуально, но для реализации не хватает точных правил по каждому компоненту.

P0Fᵢ — как сырые данные превращаются в балл 0–100

Есть:«Fᵢ = балл по фактору (0–100)».

Нет:Самого главного — функции нормализации. Как из COA/теста получить число QEF? Как из IoT-логов CCF? Как из набора отзывов CVF? Без этого Score невозможно вычислить.

Рекомендация:Для каждого из 9 факторов задать формулу/рубрику преобразования сырых сигналов в 0–100 с примерами.
P0Нет полной матрицы весов роль × фактор

Есть:Веса Wᵢ указаны в карточках части ролей (напр. «QEF 40%, PCF 25%…»).

Нет:Единой проверенной матрицы 22 роли × 9 факторов с гарантией ΣWᵢ=1.0 для каждой; для ряда ролей веса не приведены вовсе; не указано, кто и как утверждает/меняет веса.

Рекомендация:Свести единую таблицу весов, провалидировать суммы, привязать к governance-процессу изменения (см. §07).
P1«Профильный субскор» Sub формально не определён

Есть:Vᵢ = 0.40 + (Sub/100)·0.55; перетекание по релевантному фактору; λ_blind=0.85; таблица тиров→Vᵢ.

Нет:Точного определения, что такое «субскор по фактору» (это Fᵢ контрагента? нормированный вклад? как агрегируется по нескольким партиям?), и как он соотносится с 30-дневным скользящим средним численно.

Рекомендация:Дать формальное определение субскора по фактору и пример расчёта Vᵢ end-to-end.
P1Каталог штрафов Pₛ неполон, пороги серьёзности — словесные

Есть:Часть штрафов с величинами/сроками, агрегация 1−∏(1−pₖ), жёсткий потолок тира, sample-swap −0.40; уровни алерта INFO/WARNING/CRITICAL.

Нет:Исчерпывающего каталога нарушений с величинами и сроками; численных порогов, отделяющих INFO/WARNING/CRITICAL; точных «потолков тира» по каждому типу умысла.

Рекомендация:Завершить таблицу Pₛ и задать численные пороги серьёзности и потолков.
P1Мульти-оракульное требование не сведено с формулой

Есть:«Факторы с весом ≥20% требуют 2 независимых источника».

Нет:Как именно это влияет на Fᵢ/Vᵢ (обнуление? кэп? понижающий множитель?) при одном источнике; что считается «независимым».

Рекомендация:Формализовать правило (напр. кэп фактора без второго источника) и критерий независимости.
P2Параметры доминирующего фактора и New/Provisional численно мягкие

Нет:Точного порога «двух близких лидеров» (указано «~10%»), величины и срока временного буста New/Provisional, точной привязки decay к типам данных (тест vs отзыв vs IoT — сейчас одна таблица на всё).

Рекомендация:Зафиксировать численные пороги; рассмотреть разные кривые decay по типам данных.
P1Нет сквозного worked example

Нет:Полного численного примера расчёта Score одного участника от сырых данных до итогового TS и подписи доминирующего фактора — это и валидация модели, и спецификация для разработки.

Рекомендация:Добавить 1–2 эталонных расчёта (вендор, логист) как «золотой тест» для реализации.
05 — Данные и идентичность

Модель данных, KYB/KYC, верификация артефактов

P0Нет модели данных (сущности и связи)

Нет:Перечня сущностей (Участник, Профиль, Роль, Связь/Партия, COA, Тест, Отзыв, Артефакт покупки, Оракул, Инцидент, Апелляция, Событие-хеш) и их атрибутов/связей. Это фундамент схемы БД и API.

Рекомендация:Составить ER-модель с ключевыми атрибутами и кардинальностью связей.
P0KYB/KYC: провайдер и совмещение с псевдонимностью не решены

Есть:«Один участник — один профиль», KYB/KYC как замена депозита; псевдонимная верифицированная идентичность для неформальных продавцов.

Нет:Кто провайдер KYB/KYC, что именно проверяется по ролям, как серый бизнес проходит KYB (ключевой риск принятия!), как псевдонимность сочетается с «реальной юр. идентичностью» из анти-сговора, что с физлицами без юрлица.

Рекомендация:Описать уровни верификации идентичности по ролям и провайдера; провалидировать готовность серых игроков к KYB на интервью.
P1Верификация покупки и lot-identity — без технической схемы

Есть:Артефакты: трекинг, подписанный QR партии, фото флакона с лот-номером; сверка с COA.

Нет:Как именно сверяется трекинг с перевозчиком (какие интеграции/API), схема подписи QR (кто эмитент ключей лота), защита фото от повторного использования/подделки, кто генерирует лот-номера.

Рекомендация:Описать схему lot-QR (PKI), список интеграций перевозчиков и анти-реюз для медиа-доказательств.
P0Конфликт «право на удаление» ↔ неизменяемость не доспецифицирован

Есть:«On-chain только хеши; связи и идентичности off-chain с правом удаления».

Нет:Что происходит со Score и историей при удалении данных участником (пересчёт? замораживание? остаётся ли хеш-«призрак»?), как это согласуется с аудитом и с GDPR-правом на забвение (см. §09).

Рекомендация:Описать политику удаления/ретеншена и её влияние на Score и аудит-трейл.
06 — Блокчейн, крипто, ZK

Ончейн-слой, ключи оракулов, ZK-доказательства

P1Конкретика L2, что хешируется и кто платит газ

Есть:«Ethereum L2», хеши событий пересчёта и меток времени.

Нет:Выбора конкретного L2 и обоснования, схемы данных хеша, есть ли смарт-контракты или только anchoring, throughput, и — кто оплачивает газ (это вопрос финансирования, §07).

Рекомендация:Зафиксировать выбор сети, модель anchoring и носителя расходов на газ.
P1PKI оракулов: выдача и ревокация ключей лабораторий

Есть:«Криптографически подписанные COA», сверка QR с on-chain хешем.

Нет:Кто выдаёт лаборатории ключ-подписант, как происходит ревокация при компрометации/потере статуса оракула, как отзывается доверие к ранее подписанным COA.

Рекомендация:Описать жизненный цикл ключей оракулов (выдача, ротация, ревокация).
P1ZK-proof — реализуемость и стоимость не оценены

Есть:Концепт ZK-style: «доказать субскор ≥ X, не называя поставщика».

Нет:Конкретной схемы, кто генерирует/проверяет доказательство, его стоимости и латентности, есть ли это в MVP или позже. Сейчас это самая технически рискованная заявка.

Рекомендация:Оценить ZK как R&D-трек; для MVP допустить упрощение (доверенный аттестатор реестра вместо полноценного ZK).
07 — Управление и финансирование

Самый недораскрытый стратегический слой

P0Кто оператор и как обеспечивается нейтральность институционально

Есть:«Нейтральность как актив», «реестр не берёт денег ни с одной стороны».

Нет:Кто юридически управляет реестром (фонд/НКО/компания), как структура собственности гарантирует отсутствие конфликта, кто устанавливает и меняет параметры алгоритма (веса, штрафы, пороги) и как это защищено от захвата.

Рекомендация:Описать модель управления (орган, процесс изменения параметров, версионирование алгоритма, публичный changelog).
P0Модель финансирования самого реестра не определена

Есть:Честно отложено: «финансирование — на фазе 4, если понадобится для масштаба»; платная инфраструктура — опционально позже.

Нет:Кто платит за хостинг, разработку, газ L2, KYB-провайдера, арбитров, mystery-shopping — даже на MVP. «Не берём денег с оцениваемых» ≠ «нет расходов». Без базовой модели устойчивости PRD неполон.

Рекомендация:Определить источник финансирования MVP (грант/фонд/донат/B2B-API для непредвзятых потребителей отчётов — модель Carfax «платит тот, кто принимает решение»), не нарушающий нейтральность.
P1Отбор и оплата внешних арбитров (обсуждено, но не в документе)

Есть:«Обязателен внешний независимый арбитр без коммерческой ставки».

Нет:В документе — механизма формирования пула, жребия (VRF), скрининга конфликтов, отвода сторон, вознаграждения из нейтрального фонда. (Предложение проработано в чате, но не внесено в v2.)

Рекомендация:Внести в раздел 07 механизм «пул → VRF-жребий → скрининг → отвод → оплата вне исхода → подотчётность governance».
08 — Безопасность и анти-фрод

За пределами экономических атак на Score

P1Прикладная модель угроз отсутствует

Есть:Глубокая защита от гейминга Score (Sybil, сговор, sample-swap, селективное раскрытие, спам отзывов/апелляций).

Нет:Классической AppSec-модели: аутентификация/авторизация в ЛК, защита API на запись, анти-DDoS для бейджа/каталога, защита форм от спама, секьюрность хранилища off-chain, аудит доступа.

Рекомендация:Провести STRIDE-моделирование угроз приложения и описать контрмеры.
P2Граф-детектор сговора: ложные срабатывания и разбор

Нет:Порогов детектора, доли ложных срабатываний, процедуры ручного разбора и апелляции «помеченного кластера» (легальная плотная цепочка может выглядеть как кольцо).

Рекомендация:Описать пороги, human-in-the-loop и право кластера на апелляцию.
09 — Приватность и комплаенс

Юридический слой — почти не покрыт

P0Юридическая ответственность за рейтинг в серой/нелегальной зоне

Есть:«Не превращать граф в карту для прокурора», Verified-Legal сегментация, RUO-формулировки.

Нет:Анализа рисков: ответственность за публичный низкий Score (клевета/деловая репутация), правовой статус ведения реестра о потенциально нелегальной деятельности, юрисдикция оператора, дисклеймеры, политика «не консультируем по применению».

Рекомендация:Отдельный раздел правовых рисков и выбор юрисдикции; юридическая экспертиза до запуска.
P1GDPR/персональные данные vs неизменяемость

Нет:Как обрабатываются ПДн участников и покупателей (отзывы, идентичности), право на забвение против on-chain хешей, согласия, политика приватности, трансграничная передача.

Рекомендация:Описать data-protection-режим (роль контролёра/процессора, минимизация, off-chain удаление, хеши без ПДн).
P2Платёжные attestation и whistleblower — правовые нюансы

Нет:Регуляторного статуса опциональных платёжных attestation; правовых рисков системы доносов (whistleblower) и защиты заявителей.

Рекомендация:Зафиксировать как открытые юридические вопросы к модулям.
10 — UX и интерфейсы

Экраны, информационная архитектура, состояния

P1Нет описания экранов и потоков

Есть:Перечислены поверхности: каталог, профиль, ЛК/дашборд, API, бейдж, портал апелляций, виджеты, сравнение.

Нет:Информационной архитектуры, вайрфреймов, состояний (пустой/New-Provisional/under-watch профиль, вид бейджа по тирам), конструктора бейджа, экрана сравнения, портала апелляций как UI.

Рекомендация:Сделать карту экранов + low-fi вайрфреймы ключевых поверхностей MVP.
P2Локализация и доступность

Нет:Языков (ru/en?), требований доступности (a11y), мобильного поведения каталога/бейджа.

Рекомендация:Зафиксировать языки запуска и базовые a11y-требования.
11 — Метрики и аналитика

KPI, North Star, антифрод-метрики

P0Нет системы метрик успеха

Есть:Оценки виральности бейджа (показы/CTR), идея «потока обращений» как измеримой ценности.

Нет:North-Star-метрики и дерева KPI: рост участников, плотность Chain Pull-связей, % полностью верифицированных цепочек, объём и время разрешения апелляций, доля верифицированных отзывов, антифрод-показатели, удержание. Нет и определения «доверия/нейтральности» как измеримой величины.

Рекомендация:Определить North Star (напр. «кол-во верифицированных связей в графе») и дерево метрик по фазам.
12 — Операции и интеграции

Онбординг оракулов, SLA, внешние зависимости

P1Онбординг лабораторий-оракулов как процесс не описан

Есть:«API-партнёрства с лабораториями», статус Verified Oracle как маркетинг.

Нет:Процесса аккредитации оракула, договорной части, API-спеки публикации результатов, что делать, если лаборатория не хочет интегрироваться по API (ручной ввод?), bootstrap первых данных.

Рекомендация:Описать аккредитацию оракула и fallback-ввод данных без API.
P2Внешние зависимости и SLA не перечислены

Нет:Списка зависимостей (KYB-провайдер, API перевозчиков, IoT-вендоры, L2-инфраструктура) и их рисков; SLA самого реестра, инцидент-менеджмент, модерация, штат поддержки апелляций.

Рекомендация:Составить реестр зависимостей с рисками и планом на отказ каждого.
13 — Стратегические риски

Нерешённые вопросы, ставящие модель под вопрос

  • Финансирование без монетизации (P0). Нейтральность требует не брать денег с оцениваемых, но расходы (газ, KYB, арбитры, хостинг) реальны уже на MVP. Нужна модель устойчивости, не нарушающая нейтралитет.
  • Готовность серого рынка к KYB/идентичности (P0). Вся анти-Sybil и анти-сговор логика опирается на реальную идентичность, но ниша ценит анонимность. Это центральное противоречие принятия — требует валидации на интервью с реальными игроками.
  • Юридическая ответственность за рейтинги (P0). Публичная оценка участников серой/нелегальной зоны несёт риски (клевета, содействие, юрисдикция). До запуска нужна юр-экспертиза.
  • Cold start данных и оракулов (P1). Ценность реестра ≈ нулю до достижения плотности; нужен явный план первичного наполнения (откуда первые Score, первые лаборатории).
  • Мотивация арбитров без денег (P1). Качественный независимый разбор требует экспертов; чисто волонтёрская модель хрупка.
  • Реализуемость ZK в заявленном виде (P1). Самая технически амбициозная часть; риск, что в MVP придётся упростить.

15 — Рекомендуемая структура PRD

Что собрать, чтобы PRD стал исчерпывающим

Документ v2 закрывает разделы «Видение/Принципы/Механика доверия». Поверх него для PRD рекомендую добавить:

  1. Проблема, цель, North Star, метрики успеха (§01, §11).
  2. Скоуп MVP: in/out, гео, якорные роли, критерии приёмки по фазам (§01).
  3. Персоны и user stories с journey maps (§02).
  4. Функциональная спецификация: каталог, профиль, ЛК, формы, бейдж, API (§03).
  5. Алгоритм как инженерная спека: нормализация Fᵢ, матрица весов, субскор, полный Pₛ, worked example (§04).
  6. Модель данных + идентичность + верификация артефактов (§05).
  7. Ончейн/крипто-спека: L2, PKI оракулов, ZK-трек (§06).
  8. Управление + модель финансирования + арбитраж (§07).
  9. Безопасность (модель угроз) и приватность/комплаенс/право (§08, §09).
  10. UX: карта экранов и вайрфреймы (§10).
  11. Операции, интеграции, зависимости, риски (§12, §13).

Если двигаться по приоритету

Сначала закрыть 9 P0-вопросов — они блокируют не только PRD, но и принципиальную жизнеспособность (особенно финансирование, готовность к KYB и правовой статус). Параллельно достроить инженерную спеку алгоритма (нормализация Fᵢ + матрица весов) — это переводит сильную концепцию в техзадание. UX, метрики и операционка достраиваются быстро, как только зафиксированы P0/P1-решения.
