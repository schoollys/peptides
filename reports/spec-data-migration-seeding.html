Data Migration & Seeding Spec

Как наполняются данные: миграции схемы, импорт unclaimed-профилей, эталонные датасеты для регрессии скоринга и dev-фикстуры. Опирается на DB Schema и консервативную политику unclaimed (P0).

02 — Миграции схемы
  1. Forward-only, expand→contract: сначала добавляем колонки/таблицы, переключаем код, потом удаляем старое.
  2. Версионируемые файлы миграций; авто на staging, approve на prod.
  3. Обратимость для откатываемых шагов; необратимые (drop) — отдельным релизом после стабилизации.
  4. Партиционирование score_events/factor_inputs по времени при росте.
Связь с algo_version

Миграции данных НЕ пересчитывают исторические Score. Изменение параметров — новая algo_version; старые события остаются под своей версией (forward-only).

03 — Импорт unclaimed-профилей (консервативно)
ПравилоРеализация
Только нейтральные фактыИз легитимных публичных источников; никаких негативных/репутационных Score
Статусunclaimed; полноценный TS не считается до claim+верификации
Возражение/удалениеЯвный механизм: участник может оспорить/удалить до claim
ПроисхождениеПоле источника данных; аудируемость импорта
ДедупликацияПо домену/имени (trgm) перед вставкой

Импорт — управляемый пайплайн с ревью, не массовый скрейп. Юридическую рамку (вне scope этого пакета) утверждает counsel.

04 — Golden-датасеты (регрессия скоринга)
НаборСодержание
GD-basicТиповой вендор по всем 9 факторам → ожидаемый Score (worked example)
GD-edgeГраницы Fᵢ (0/100), missing data, balanced, no-TS, provisional
GD-propСвязи: одиночная, концентрация >0.6, blind (λ), цикл (запрет), мульти-источник (max)
GD-penaltyОдин/несколько флагов, CRITICAL-потолок тира, истёкший флаг
GD-decayВозраст наблюдений по классам A–D, проверка floor
Использование

Каждый набор привязан к algo_version. CI прогоняет их при каждом изменении ядра; расхождение = красный гейт (Test Strategy).

05 — Dev-фикстуры (seed)
  1. Участники по всем 22 ролям, разные тиры/статусы (unclaimed/provisional/active/suspended/tombstone).
  2. Оракулы с dev-сертификатами; один «отозванный» (для CRL-тестов).
  3. Лоты + COA: валидные и заведомо битые (custody incomplete, media reuse).
  4. Связи контрагентов, включая заведомый цикл и кластер «сговора».
  5. Флаги Pₛ разных severity; апелляция в каждом статусе.
  6. B2B-подписка + локальный webhook-sink.

Команда: pnpm db:seed (см. Dev Onboarding).

06 — Сидирование параметров / algo_version
  1. Засеять algo_versions (version=1, is_current=true).
  2. Загрузить role_factor_weights (матрица 22×9) и полосы факторов из Parameters Registry.
  3. Зафиксировать decay-классы, λ_blind, cap, пороги тиров.
  4. При SME-апруве (v0.9→v1.0) — новая версия, без пересчёта истории.
