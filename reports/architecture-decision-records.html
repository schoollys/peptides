Architecture Decision Records

Зачем зафиксированы ключевые технические решения с обоснованием и последствиями — чтобы через полгода команда не переписывала всё «потому что забыли почему». Формат каждой записи: Контекст → Решение → Обоснование → Последствия → Рассмотренные альтернативы. Все 15 решений зафиксированы (статус Accepted). Открытыми осознанно остаются лишь конкретные точки выбора — финальная L2-сеть (Base vs Arbitrum), провайдеры KYB и KMS/HSM, облачный провайдер, состав хешируемого payload и триггеры масштабирования — они помечены в разделах «Последствия» и привязаны к внешним данным/зависимостям (см. Integration Map, реестр §14 PRD).

ADR-001Модульный монолит firstAccepted
Контекст

Lean-команда, MVP, транзакционный скоринг, неопределённый профиль нагрузки.

Решение

Модульный монолит для MVP/V1; вне ядра только async-worker и signing-сервис.

Обоснование
  • Минимум операционной сложности при lean-команде
  • ACID для скоринга в единой БД
  • Модульные границы = будущие швы для выделения сервисов
Последствия

Нужна дисциплина границ модулей (нет прямого доступа в чужие таблицы). Масштабирование частей — через реплики/воркеры до выделения сервисов.

Альтернативы

Микросервисы — отклонено (преждевременная сложность); единый немодульный монолит — отклонено (нет швов для эволюции).

ADR-002PostgreSQL как первичное хранилищеAccepted
Контекст

Реляционная модель (Participant↔ScoreEvent↔FactorInput↔Lot↔COA), потребность в ACID и сложных выборках для каталога/ранжирования.

Решение

PostgreSQL — основная БД; Redis — кэш и очередь; Object Storage — блобы.

Обоснование
  • Сильная реляционная модель и транзакции для скоринга
  • Зрелость, расширения (JSONB для гибких полей, full-text для поиска)
  • Простота при lean-эксплуатации
Последствия

Шардинг/партиционирование при росте (по диапазону id). Тяжёлую аналитику B2B — выносить в реплику/витрину.

Альтернативы

NoSQL (Mongo) — отклонено (нужны транзакции/связи); сразу OLAP — преждевременно.

ADR-003Хостинг и data-residency в EU/CHAccepted
Контекст

Правовой режим CH/EU (Волна 0), GDPR, обработка KYB/ПДн.

Решение

Прод-инфраструктура и хранение данных — в EU/CH регионах; провайдер с соответствием GDPR.

Обоснование
  • Соответствие выбранному правовому режиму и DPIA
  • Минимизация трансграничной передачи ПДн (SCC только где неизбежно)
Последствия

Выбор провайдера ограничен регионом; US-аудитория обслуживается через CDN-edge, но первичные данные — EU/CH.

Альтернативы

US-хостинг — отклонено (расходится с правовым режимом и усложняет GDPR).

ADR-004L2 Ethereum (Base/Arbitrum) через HashAnchorAccepted
Контекст

Нужна публичная неизменяемая фиксация целостности без хранения данных в цепи; стоимость газа важна.

Решение

Анкоринг хешей в Ethereum L2 (Base/Arbitrum) через абстракцию HashAnchor с Merkle-батчингом.

Обоснование
  • Низкая стоимость газа L2 vs L1; экосистема и надёжность Ethereum
  • Абстракция HashAnchor отвязывает от конкретной сети (смена L2 без переписывания)
  • Merkle-батчинг минимизирует число транзакций
Последствия

Зависимость от RPC-провайдера/нод L2 (см. Integration Map). Финальный выбор Base vs Arbitrum — за бенчмарком стоимости/латентности.

Альтернативы

L1 Ethereum — отклонено (дорого); приватный блокчейн — отклонено (нет публичной проверяемости); другие L1 — отклонено (меньше доверия/экосистемы).

ADR-005Данные off-chain, в цепь — только анонимные хешиAccepted
Контекст

GDPR/право на удаление несовместимо с записью ПДн в неизменяемую цепь.

Решение

Все данные — off-chain (PostgreSQL/Storage); в L2 идут только hash(payload‖salt) без идентификаторов; удаление — tombstone.

Обоснование
  • Совместимость с правом на удаление (off-chain удаляемо)
  • Анонимные хеши = не ПДн (соль, без идентификаторов) — подтверждает DPO
  • Целостность сохраняется (хеш сверяется)
Последствия

Состав каждого payload утверждается DPO на анонимность. Восстановление по хешу невозможно (by design).

Альтернативы

Полные данные on-chain — отклонено (нарушает GDPR, дорого).

ADR-006Бейдж — серверный подписанный SVG/PNGAccepted
Контекст

Бейдж встраивается на любые сайты; нужна совместимость и защита от подделки значения.

Решение

Server-rendered подписанное изображение через signed endpoint; не JS-виджет.

Обоснование
  • Работает без JS на любой площадке (широкая совместимость)
  • Подпись эндпоинта = защита от подмены значения
  • Кэшируемо на CDN (производительность)
Последствия

Интерактивность ограничена (клик→профиль); «живость» — через TTL-кэш и пересчёт.

Альтернативы

JS-виджет — отклонено (блокируется, подделываем, тяжелее); статичная картинка без подписи — отклонено (подделка).

ADR-007KYB/KYC через стороннего провайдераAccepted
Контекст

Верификация бизнеса/личности — регуляторно сложна и не является нашим ядром ценности.

Решение

Интеграция стороннего KYB/KYC-провайдера; мы храним только статус/уровень, не сырьё (provider — процессор).

Обоснование
  • Не изобретаем комплаенс-ядро; быстрее к рынку
  • Минимизация хранимых ПДн (privacy-by-design)
Последствия

Зависимость от покрытия провайдера по юрисдикциям → fallback-механика (BR-KYB-FALLBACK). Выбор провайдера — P1 (Integration Map).

Альтернативы

Собственный KYB — отклонено (дорого, не ядро, риск).

ADR-008Ключи в KMS/HSM, выделенный signing-сервисAccepted
Контекст

Подпись бейджей, COA-верификация, anchoring — критичные крипто-операции; компрометация ключа = катастрофа доверия.

Решение

Приватные ключи — только в KMS/HSM; доступ через изолированный signing-сервис; ротация и CRL.

Обоснование
  • Ключи не покидают защищённый модуль
  • Изоляция сервиса ограничивает поверхность атаки
  • Поддержка ротации/отзыва (PKI-lifecycle)
Последствия

Доступ к подписи — узкое место с аудитом; провайдер KMS/HSM — зависимость (Integration Map).

Альтернативы

Ключи в приложении/файлах — отклонено (риск утечки).

ADR-009TypeScript / Node для продуктаAccepted
Контекст

Web-продукт, SSR, единый язык фронт+бэк, богатая экосистема для крипто/web3/SVG.

Решение

TypeScript на Node (бэкенд + SSR); типобезопасность сквозь стек. Финализировано как стандарт стека.

Обоснование
  • Единый язык фронт/бэк → меньше контекста для lean-команды
  • Сильные web3/crypto-библиотеки, SVG-рендеринг
  • Типобезопасность контрактов API
Последствия

CPU-тяжёлый скоринг — выносить в воркер; при необходимости — отдельный сервис на более «числовом» рантайме.

Альтернативы

Python (хорош для аналитики, но два языка), Go (быстрее, но меньше web3-экосистемы для full-stack lean). Отклонены в пользу единого TS-стека.

ADR-010Гибрид: событийный пересчёт + суточный batchAccepted
Контекст

Алерты не должны опаздывать, но пересчёт decay для всех в реальном времени избыточен (BR-CADENCE).

Решение

Критичные события (Pₛ-флаг, новый тест, отмена демпфера) — пересчёт сразу; decay/пропагация — batch раз в сутки.

Обоснование
  • Оперативность защитных алертов + разумная стоимость инфраструктуры
  • Decay предсказуем, не требует реального времени
Последствия

Нужны очередь и идемпотентные воркеры; согласованность кэша после batch.

Альтернативы

Всё в реальном времени — отклонено (дорого); всё батчем — отклонено (алерты опаздывают).

ADR-011MVP: shielded assertion, полный ZK — в V2Accepted
Контекст

Blind-раскрытие требует доказательства факта без раскрытия личности; полноценный ZK сложен и дорог в реализации.

Решение

MVP/V1 — «shielded assertion» (реестр как доверенная третья сторона); полный ZK — V2.

Обоснование
  • Быстрее к рынку без ZK-инженерии
  • Достаточный уровень приватности для старта при доверии к оператору
  • Чёткий путь апгрейда до ZK без смены семантики
Последствия

До V2 приватность опирается на доверие к оператору (гибридное управление, Волна 0). Миграция на ZK спланирована.

Альтернативы

Сразу полный ZK — отклонено (срок/стоимость); без приватности — отклонено (нужно посредникам).

ADR-012Frontend — Next.js (React, SSR/RSC)Accepted
Контекст

Каталог и профили должны быть SEO-индексируемы и быстро отдаваться (read-path SLO); бейдж — отдельный серверный эндпоинт (ADR-006).

Решение

Next.js (React) с SSR/RSC как фронтенд-фреймворк на Node/TypeScript.

Обоснование
  • SSR/SEO «из коробки» для публичного каталога и профилей
  • Единый TS-стек (ADR-009), общие типы контрактов с бэкендом
  • Зрелая экосистема, кэш/ISR для read-тяжёлых страниц
Последствия

Граница между SSR-слоем Next и API NestJS должна быть чёткой; рендеринг бейджа — вне Next (выделенный signing-сервис, ADR-006/008).

Альтернативы

Remix (меньше распространён в команде), SPA без SSR — отклонено (SEO/латентность каталога).

ADR-013Backend API — NestJS поверх FastifyAccepted
Контекст

Модульный монолит из 13 доменных модулей с чёткими границами (ADR-001); нужны DI, инкапсуляция модулей, единый формат ошибок/валидации.

Решение

NestJS (DI, модули) с Fastify-адаптером в качестве HTTP-слоя.

Обоснование
  • Модули NestJS ложатся 1:1 на доменные границы — главная страховка монолита
  • DI/провайдеры упрощают вынос модуля в сервис при эволюции
  • Fastify даёт производительность read-пути; единый pipeline валидации/ошибок (API Conventions)
Последствия

Дисциплина границ модулей контролируется линтером (Coding Standards); чистое скоринг-ядро остаётся фреймворк-независимым.

Альтернативы

Голый Fastify (легче, но без структуры под 13 модулей), Express (медленнее) — отклонены.

ADR-014Асинхронные задачи — BullMQ на RedisAccepted
Контекст

Гибрид «событие + суточный batch» (ADR-010): пересчёт Score, пропагация, anchoring, transactional outbox/вебхуки — нужны очереди и идемпотентные воркеры.

Решение

BullMQ поверх уже имеющегося Redis (ADR-002) для очередей и планировщика.

Обоснование
  • Не вводит новую инфраструктуру — Redis уже в стеке
  • Ретраи/backoff/повторяемые задания «из коробки» (anchoring, вебхуки)
  • Идемпотентность через job-id и transactional outbox
Последствия

Redis становится критичным для воркеров — нужны HA и мониторинг глубины очереди (Observability).

Альтернативы

Выделенный брокер (RabbitMQ/Kafka) — избыточен для MVP-нагрузки; managed-очередь облака — лок-ин, отложено.

ADR-015Оркестрация — managed-контейнеры (k8s-ready), IaC — TerraformAccepted
Контекст

EU/CH-хостинг (ADR-003), lean-команда, потребность в воспроизводимой инфраструктуре без преждевременной сложности.

Решение

Старт на управляемом контейнер-сервисе (Docker-образы), архитектурно k8s-ready; вся инфраструктура — через Terraform.

Обоснование
  • Минимум операционной нагрузки на старте, путь миграции в k8s по росту
  • Terraform: версионируемые сеть/БД/кэш/storage/секреты (Infra & DevOps)
  • Контейнеризация даёт паритет dev/stage/prod
Последствия

Двенадцатифакторное приложение; конфиг — из окружения/секрет-менеджера, не в образе.

Альтернативы

Сразу self-managed k8s — отклонено (оверхед для MVP); чистый PaaS без IaC — отклонено (воспроизводимость/аудит).
