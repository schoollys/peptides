# ADR — Ролевая модель MVP: VENDOR-rename, RETAILER-якорь и отдельный MANUFACTURER

Статус: Accepted (2026-06-23) · Развивает реализацию «3 якорные роли + MANUFACTURER».

## Обновление (2026-06-26): P0-калибровка скоринга (algo БД `v2.6.0`, движок `v1.3.0`)

Проблема: верифицированные игроки тонули в `Watch` из-за двух дефектов.
1. **Класс decay брался из числа дней** (`diClass(days)` в генераторе), хотя по
   спеке §04 класс — это свойство **типа доказательства**, а не его возраста.
   Введён канонический `FACTOR_DECAY_CLASS` в движке: QEF/PCF/CCF/TRF → A (365д,
   регуляторика медленно устаревает), SCIF/RF → B (120д), FRF → C (45д,
   логистика), CVF/CVF_B → D (90д). Генератор и БД-путь (`recompute-all`) теперь
   используют его; `updated_days_ago` остаётся фактическим возрастом.
2. **KYB не влиял на Score** (был лишь бейджем). Введён `KYB_VI_FLOOR`
   (L0 0 / L1 .5 / L2 .65 / L3 .8): `Vᵢ = max(Vᵢ, floor[level])` по каждому
   фактору, не выше cap 0.85. Юридическая верификация теперь поднимает доверие.

`ComputeInput.kybLevel` — новый опциональный вход; покрыто тестами (28/28).
Эффект: Lonza 31.6→65.1 (Bronze→Gold), Adragos 17.1/Watch→49.2/Silver, топовые
CDMO/заводы — Gold, Bachem — Platinum; непроверенные PROVISIONAL-витрины остаются
в Watch. `algo_version` БД поднята `v2.5.0 (4) → v2.6.0 (5)` (forward-only).

## Обновление (2026-06-25): добавлена якорная роль FILL_FINISH (CDMO)

Между `MANUFACTURER` (синтез bulk-API) и `VENDOR`/`RETAILER` (точка покупки) в
реальной цепочке стоит **контрактный фасовщик / CDMO**: асептический розлив
bulk-API во флаконы и **лиофилизация** с особыми условиями хранения. Это частая
точка подмены/деградации, поэтому роль введена как **отдельная якорная**.

| Роль | Кто | Сторона | Buyer-facing | Якорь |
|---|---|---|---|---|
| `FILL_FINISH` | Фасовщик / CDMO (розлив + лиофилизация) | midstream | нет (B2B) | да |

- Матрица `FILL_FINISH` — quality/process-heavy: `QEF .35 / PCF .30 / SCIF .20 /
  CVF_B .15` (ΣWᵢ=1). `SCIF` привязывает финишный флакон к исходному API-лоту;
  `CVF_B` — верифицированные B2B-подтверждения (бренд/вендор).
- `ALGO_VERSION` поднят `v1.1.0 → v1.2.0` (добавление роли; существующие роли и их
  Score **не изменились** — forward-only). Версия БД (`v2.5.0`) не трогается: в
  данных пока нет участников с ролью `FILL_FINISH`, новых score_events не создано.
- Итог: **8 ролей, 6 якорных** (`MANUFACTURER, FILL_FINISH, VENDOR, LABORATORY,
  DISTRIBUTOR, RETAILER`). Роль автоматически появляется в фильтре каталога
  (`ROLE_OPTIONS` строится из `ROLE_LABELS` + `ANCHOR_ROLE_CODES`).
- Связанные изменения: `packages/core-scoring/src/{types,params}.ts`,
  `lib/participants.ts`, `scripts/{seed,gen-participants-from-csv}.ts`.

## Обновление (2026-06-23, поздн.): возврат MANUFACTURER как отдельной upstream-роли

После решения «VENDOR = витрина/бренд» появилась потребность отдельно ранжировать
**чистые синтез-заводы** (B2B API-producers, не buyer-facing). Поэтому `MANUFACTURER`
возвращён как **отдельная якорная роль**, теперь семантически не конфликтует с `VENDOR`:

| Роль | Кто | Сторона | Buyer-facing | Якорь |
|---|---|---|---|---|
| `MANUFACTURER` | Завод-синтез / API-producer | upstream | нет (B2B) | да |
| `VENDOR` | Витрина/бренд, у которого покупают | downstream | да | да |

- Матрица `MANUFACTURER` — synthesis-quality heavy: `QEF .3 / PCF .2 / SCIF .2 /
  TRF .15 / CCF .1 / CVF .05` (worked example `spec-algorithm §12`).
- `VENDOR` веса в MVP **не менялись** (избежать массового пересчёта); их сдвиг к
  fulfilment (FRF/CVF) + наследование качества от `MANUFACTURER` через
  `counterparty_links` (Vᵢ-пропагация) — **V1**.
- `ALGO_VERSION` поднят `v1.0.0 → v1.1.0` (добавление роли; существующие роли и их
  Score не изменились — forward-only).
- Итог: **7 ролей, 5 якорных** (`MANUFACTURER, VENDOR, LABORATORY, DISTRIBUTOR,
  RETAILER`). Связь vendor/retailer → manufacturer даёт цель для V1-пропагации,
  снимающей потолок ритейлера при раскрытии верифицированного источника.
- Текущие витрины `p-002 / p-005 / p-007` остаются `VENDOR`; добавлен образцовый
  завод-синтез `p-013` (NordSynth API GmbH).
- **Граф контрагентов засеян** (`counterparty_links`, направление downstream→upstream):
  `p-002/p-005/p-007 (VENDOR) → p-013` и `p-009 (RETAILER) → p-013` (blind-линк,
  `λ=0.85`). Источник   истины — `COUNTERPARTY_LINKS` в `lib/participants.ts`. Граф визуализирован на
  профиле (блок «Цепочка поставок»: «Закупает у» / «Поставляет», доля лотов и
  бейдж blind-линка).

## V1 (2026-06-23): Vᵢ-пропагация + условный потолок роли (algo `v2.5.0`)

- **Vᵢ-пропагация в `recompute-all`.** Для каждого участника по исходящим
  (downstream→upstream) рёбрам собираются субскоры (Fᵢ) верифицированного
  источника и применяется `propagatedVi(base, edges)` по каждому фактору.
  Правило **no-harm**: `Vᵢ = max(base, propagated)` — напрямую верифицированный
  узел (Vᵢ выше cap 0.85) никогда не штрафуется; пропагация только поднимает Vᵢ
  узлам, опирающимся на источник.
- **Условный потолок роли.** В движок добавлен `ComputeInput.upstreamVerified`;
  `ROLE_TIER_CEILING` (RETAILER→Gold) снимается, если у реселлера есть upstream-связь
  с юридически верифицированным производителем. CRITICAL-хардкап при этом не
  обходится. Покрыто тестами.
- **Версия.** `algo_version` БД поднята до **v4 = `v2.5.0`** (forward-only); прежние
  score_events остаются под своими версиями.
- **Эффект на текущих данных — почти нулевой и это ожидаемо:** витрины уже
  верифицированы выше cap (Vᵢ 0.91–0.96), ритейлер `p-009` (25.1, Watch) — ниже
  потолка Gold и его единственное blind-ребро попадает под concentration-penalty.
  Заметно сдвинулся лишь `p-005` (42.4→42.6). Это фундамент: механизм включён и
  версионирован, видимый эффект появится после калибровки данных/весов.
- **Buyer-facing перевес `VENDOR` (сделано).** Витринам засеян фактор `FRF`
  (fulfilment), веса `VENDOR` сдвинуты к доставке при сохранении качества:
  `FRF .30 / QEF .25 / PCF .20 / SCIF .15 / TRF .10` (ΣWᵢ=1). `CVF` (buyer
  confirmations) пока вне весов — собран не у всех витрин; добавить после
  выравнивания покрытия (TODO SME). Эффект: `p-005` 42.6→49.1, `p-007` 54.1→57.5,
  `p-002` 85.0 (Platinum) — витрины теперь оцениваются с упором на fulfilment.
  Worked-example из §12 (production-матрица) переадресован на роль `MANUFACTURER`.

## (Исходное решение, ниже — для истории)

## Контекст

Полная таксономия рынка — 22 роли в 11 категориях (см. `reports/PRD.html`,
`artifacts/v2-peptide-trust-registry.html`). В коде MVP исторически жил
упрощённый срез из 6 ролей с тремя якорными (`MANUFACTURER`, `LABORATORY`,
`DISTRIBUTOR`), где `MANUFACTURER` имел лейбл «RUO-вендор».

Возражение с продуктовой стороны: в RUO-сегменте конечный покупатель почти
никогда не контактирует с заводом-синтезом — он покупает у **витрины/бренда**
или у **ритейлера/реселлера**. MVP, в котором якорные роли — это только
завод + лаборатория + GB-организатор, не закрывает основную точку входа спроса
и усиливает риск cold-start (`reports/discovery-pack.html`, риск R-S1).

Одновременно имя кода `MANUFACTURER` вводило в заблуждение: смысл якорной роли —
«вендор/бренд, у которого покупают», а не обязательно производитель.

## Решение

1. **Переименование `MANUFACTURER` → `VENDOR`** во всём коде (типы, матрица весов,
   seed, шаблоны, mock, UI). Семантика: **RUO-вендор = витрина/бренд, у которого
   покупают**; собственный синтез не обязателен.

2. **`RETAILER` поднят в якорные роли** (теперь 4 якорные:
   `VENDOR`, `LABORATORY`, `DISTRIBUTOR`, `RETAILER`). Ритейлер/реселлер — тоже
   точка покупки, поэтому buyer-facing и должен присутствовать в каталоге MVP.

3. **Потолок тира для реселлера.** Доверие чистого реселлера опирается на пока
   неверифицированный upstream-источник, а Vᵢ-пропагация между контрагентами —
   фича V1 (`reports/prd-scope-document.html` §03). Поэтому до V1:
   - `ROLE_TIER_CEILING = { RETAILER: 'Gold' }` — тир не выше **Gold**, независимо
     от числового Score (движок `packages/core-scoring`, зеркало в read-path
     `lib/repository.ts` через `capTierToRole`);
   - на профиле — дисклеймер «источник не верифицирован, полное доверие после
     Vᵢ-пропагации (V1)» (`components/profile/profile-hero.tsx`).

## Маппинг на таксономию 22 ролей

| Код MVP | Полная таксономия | Якорь | Примечание |
|---|---|---|---|
| `VENDOR` | RUO-вендор / бренд (категория «предложение») | да | витрина/бренд, у которого покупают |
| `LABORATORY` | Независимая лаборатория-оракул | да | источник Vᵢ |
| `DISTRIBUTOR` | GB-организатор / дистрибьютор | да | агрегатор закупок |
| `RETAILER` | Розница / реселлеры / дропшипперы (кат. VIII) | да | потолок Gold до V1 |
| `CONSULTANT` | Консультант | нет | non-anchor |
| `IMPORTER` | Импортёр | нет | non-anchor |

## Последствия

- Покупатель находит в каталоге те поверхности, у которых реально покупает.
- Honest-by-design: реселлер виден, но его потолок и дисклеймер не выдают
  невыверенное доверие за полное.
- Потолок снимается/смягчается, когда в V1 появится Vᵢ-пропагация: тогда Score
  реселлера начнёт наследовать верифицированный субскор upstream-источника.
